# 冒泡排序 vs 快速排序 — 时延对比报告

## 测试环境
- Python 版本: 3.14.4 (main, Apr  7 2026, 13:13:20) [Clang 21.0.0 (clang-2100.0.123.102)]
- 平台: macOS-26.5.1-arm64-arm-64bit-Mach-O
- 测试时间: 2026-06-28 12:50:32
- 每种规模运行 5 轮取平均耗时
- 数据分布: 随机（均匀分布整数，值域 [0, n)）

## 测试方法

### 冒泡排序
- 双层循环，相邻比较并交换
- 带提前终止优化：一轮无交换即判定有序
- 时间复杂度: 最好 O(n)，平均/最坏 O(n²)
- 空间复杂度: O(1) 原地排序

### 快速排序
- 分治策略，Lomuto/Hoare 变体，以首个元素为 pivot
- 平均 O(n log n)，最坏 O(n²)（但随机数据下近似平均情况）
- 空间复杂度: O(log n) 递归栈

## 详细时延对比

| 规模 | 冒泡排序 (ms) | 快速排序 (ms) | 加速比 |
|------|---------------|---------------|--------|
|   100 | 0.121 | 0.024 | 5.0x |
|   500 | 3.306 | 0.151 | 22.0x |
|  1000 | 12.768 | 0.301 | 42.4x |
|  2000 | 51.471 | 0.674 | 76.4x |
|  5000 | 326.440 | 1.764 | 185.1x |
| 10000 | 1336.456 | 3.750 | 356.4x |

> 加速比 = 冒泡排序平均耗时 / 快速排序平均耗时，值越大说明快速排序优势越明显。

## 汇总分析

### 性能差异趋势

从表中可以看出：

- **冒泡排序** 从规模 100 增长到 10000（约 100 倍），耗时增长约 11048 倍，基本符合 O(n²) 增长规律 （理论增长 10000 倍）。
- **快速排序** 同区间耗时增长约 154 倍，介于 O(n)（100 倍）和 O(n log n) （100 倍估算）之间，符合预期。

### 各规模加速比

- **n = 100**: 5.0x
- **n = 500**: 22.0x
- **n = 1000**: 42.4x
- **n = 2000**: 76.4x
- **n = 5000**: 185.1x
- **n = 10000**: 356.4x

加速比随规模呈超线性增长——冒泡的 O(n²) 与快排的 O(n log n) 差距随 n 增大而急剧拉大。

### 复杂度理论验证

| 规模 | 冒泡实测倍率 | 快排实测倍率 | O(n²) 理论倍率 | O(n log n) 理论倍率 |
|------|-------------|-------------|----------------|---------------------|
|   500 | 27.33x | 6.16x | 25.00x | 6.75x |
|  1000 | 3.86x | 2.00x | 4.00x | 2.22x |
|  2000 | 4.03x | 2.24x | 4.00x | 2.20x |
|  5000 | 6.34x | 2.62x | 6.25x | 2.80x |
| 10000 | 4.09x | 2.13x | 4.00x | 2.16x |

冒泡排序的实测增长倍率接近 O(n²) 理论值，快速排序则接近 O(n log n) 理论值，验证了复杂度分析的准确性。

## 结论

1. **快速排序在全部测试规模下显著优于冒泡排序**，且规模越大优势越明显。
2. 在最大规模 10000 时，快速排序比冒泡排序快约 356 倍。
3. 冒泡排序的 O(n²) 时间复杂度使其在 n > 2000 时基本不可用。
4. 快速排序在随机数据上表现出稳定的 O(n log n) 性能。
5. 实际生产环境推荐 Python 内置 `sorted()`（Timsort），综合性能优于手写快速排序。
