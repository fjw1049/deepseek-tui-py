🔔 第一部分：Hooks 系统

1. Hook 事件广播流程

1 事件触发 - 引擎中发生事件（ResponseStart, ToolLifecycle 等）
2 调度器发射 - HookDispatcher.emit(event) 被调用
3 序列化 - 事件序列化为 JSON 格式
4 广播 - 广播到所有已注册的 sinks（fire-and-forget）
5 独立处理 - 每个 sink 独立处理事件

2. HookEvent 类型

ResponseStart

响应开始事件

{ response_id }

ResponseDelta

响应增量事件

{ response_id, delta }

ResponseEnd

响应结束事件

{ response_id }

ToolLifecycle

工具生命周期事件

{ response_id, tool_name, phase, payload }

JobLifecycle

任务生命周期事件

{ job_id, phase, progress, detail }

ApprovalLifecycle

审批生命周期事件

{ approval_id, phase, reason }

GenericEventFrame

通用事件帧

{ frame }

3. Hook 生命周期阶段

Tool 阶段
  • • pre_execution
  • • post_execution
  • • error
Job 阶段
  • • started
  • • progress
  • • completed
  • • failed
Approval 阶段
  • • requested
  • • approved
  • • denied
  • • timeout

4. Hook Sinks

StdoutHookSink

标准输出 Sink

  • 将 JSON 打印到 stdout
  • 实时输出事件
  • 适用于调试和日志记录

JsonlHookSink

JSONL 文件 Sink

  • 追加带时间戳的事件到 JSONL 文件
  • 路径: ~/.deepseek/hooks.jsonl
  • 格式: {"at": RFC3339, "event": HookEvent}
  • 持久化事件历史

WebhookHookSink

Webhook HTTP Sink

  • HTTP POST 到 webhook URL
  • 重试逻辑: 最多 3 次尝试
  • 指数退避: 200ms * retry_count
  • 负载: {"at": RFC3339, "event": HookEvent}
  • 支持远程集成

HookDispatcher

struct HookDispatcher {
    sinks: Vec<Arc<dyn HookSink>>
}

impl HookDispatcher {
    async fn emit(&self, event: &HookEvent) {
        for sink in &self.sinks {
            sink.emit(event).await;
        }
    }
}

HookSink Trait

trait HookSink: Send + Sync {
    async fn emit(&self, event: &HookEvent);
}

// 实现
struct StdoutHookSink;
struct JsonlHookSink { path: PathBuf }
struct WebhookHookSink {
    url: String,
    client: HttpClient
}

💾 第二部分：持久化系统

5. SQLite 数据库架构

表名 字段 说明
sessions id, created_at, updated_at, workspace, model, mode 会话元数据
messages id, session_id, role, content, created_at, turn_index 消息历史
checkpoints id, session_id, created_at, snapshot_data 会话快照
queued_requests id, session_id, request_data, created_at, status 离线请求队列

索引

  • sessions(workspace) - 按工作区查询
  • messages(session_id, turn_index) - 按会话和轮次查询
  • checkpoints(session_id, created_at DESC) - 获取最新快照

6. 会话存储流程

1 创建会话 - INSERT INTO sessions
2 添加消息 - INSERT INTO messages
3 完成轮次 - UPDATE session updated_at
4 关闭会话 - UPDATE session status

7. 检查点机制

1 风险操作前 - 创建检查点
2 序列化状态 - 将会话状态序列化为 JSON
3 保存快照 - INSERT INTO checkpoints
4 崩溃检测 - 检测到非正常关闭
5 加载快照 - 加载最新检查点
6 恢复状态 - 反序列化会话状态
7 继续执行 - 从最后一轮继续

8. 崩溃恢复流程

1 启动检查 - State::open() 检查非正常关闭标志
2 查询快照 - 如果非正常关闭，查询每个会话的最新检查点
3 恢复状态 - 从 checkpoint.snapshot_data 恢复会话状态
4 重放 WAL - 如果需要，重放 Write-Ahead Log
5 恢复工具执行 - 恢复进行中的工具执行（标记为失败）
6 清除标志 - 清除非正常关闭标志

9. 离线队列

1 网络错误 - 请求因网络错误失败
2 序列化请求 - 将请求序列化为 JSON
3 入队 - INSERT INTO queued_requests with status='pending'
4 重新连接 - 检测到网络恢复
5 查询队列 - 查询 pending 状态的请求
6 重试请求 - 按顺序重试请求
7 更新状态 - UPDATE status='completed' 或 'failed'
8 去重 - 通过请求指纹去重
