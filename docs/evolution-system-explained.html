进化系统通俗全解

审核代码后的完整说明：Agent 如何从对话里「学会」下次怎么做更好，而不是只记住发生过什么。

一句话类比：Smart Memory 像图书馆员自动归档聊天记录；Evolution 像私人教练——把反复踩过的坑写成便签（策展记忆），把熟练的流程写成操作手册（技能），还会在后台默默复盘，并在会话结束前抢救重要经验。
为什么需要 与 Memory 区别 三层架构 两种资产 完整流程 两个工具 后台复盘 丢失前抢救 审批账本 注入 Prompt 界面 配置 代码审核

1为什么需要进化系统？

普通对话结束后，模型「忘掉」了这次怎么解决问题的细节。下次遇到类似任务，又要从零摸索。

Evolution 解决的是经验沉淀问题：

📝 知道什么（策展记忆）

项目惯例、用户偏好、踩过的坑——写成短条目，下次对话直接读进 system prompt。

📖 怎么做（技能）

可重复的操作流程（部署 checklist、代码审查步骤）——写成 SKILL.md，需要时 load_skill 加载。

口诀：Memory 管「知道什么」；Evolution 管「下次怎么做更好」。

2与 Smart Memory 的区别

维度Smart Memory（已有）Experience Evolution（进化）
谁写入系统自动（L0→L3 pipeline）主 Agent 主动调工具 + 后台复盘 Agent
产物结构化记忆层 L1/L2/L3MEMORY.md / USER.md + SKILL.md
注入方式recall 结果（volatile）会话级 frozen 快照 + 本 session 新技能通知
典型内容对话摘要、事实提取项目惯例、操作流程、用户画像
审批无Ledger 策略：auto / propose / deny

两者并行运行，通过 PostTurnOrchestrator 在每轮对话结束后统一调度，互不替代。

Layer 1 — PostTurn 共享运行时

每轮对话结束后，Engine 构建一份 TurnEvidence（证据包），交给编排器分发给各 pipeline：

  • 用户说了什么、用了多少工具轮次、是否成功
  • 本轮消息切片（供 capture / review 用）
  • flush_mode：压缩上下文或退出时设为 true，触发抢救逻辑

Layer 2 — Experience Backends

两种「落盘后端」：

  • CuratedMemoryBackend → ~/.deepseek/memories/MEMORY.md 和 USER.md
  • ProceduralSkillBackend → {workspace}/.deepseek/skills/{name}/SKILL.md

Layer 3 — Experience Ledger

所有写入（无论主 Agent 还是后台复盘）必须过 Ledger：策略决策 → 审计记录 → 执行或等待用户批准。

4两种资产详解

策展记忆（Curated Memory）

文件位置：

~/.deepseek/memories/MEMORY.md — Agent 笔记（项目惯例、踩坑记录）

~/.deepseek/memories/USER.md — 用户画像（偏好、约束、身份）

格式：条目用 § 分隔；MEMORY 上限约 2200 字符，USER 约 1375 字符（可配置）。

特点：会话启动时加载一次快照，写入 Engine._curated_snapshot，同一会话中途不更新 stable 层（避免 prompt 抖动）。工具写入会更新磁盘，但当前 session 的 stable prompt 不变。

程序性技能（Procedural Skills）

文件位置：

{workspace}/.deepseek/skills/{name}/SKILL.md（默认 project 作用域）

~/.deepseek/skills/{name}/（user 作用域，可选）

结构：YAML frontmatter（name + description）+ Markdown 正文；可附带 references / templates / scripts / assets 子目录。

特点：新技能创建后，会在本 session 的 <session-evolution> volatile 块里追加一行通知，提醒模型可以 load_skill。

5完整生命周期（一步不漏）

①
会话启动 Engine.create() → 若 evolution.enabled，构建 EvolutionPipeline → 加载策展快照 → PostTurnOrchestrator.start()
②
每轮对话开始 recall 记忆 → build_system_prompt（含策展快照 + 进化指引 + session-evolution 通知）→ 模型工作
③
主 Agent 主动沉淀（可选） 模型调用 memory_curate / skill_manage → Ledger 审批 → 落盘或弹出 EvolutionBubble 等待用户确认
④
Turn 结束 构建 TurnEvidence → post_turn.after_turn() → MemoryPipeline 可能 capture；EvolutionPipeline 检查是否该复盘
⑤
后台复盘（异步，用户无感） 满足条件时 create_task → 独立子 Agent 读最近 8 轮 transcript → 调 memory_curate/skill_manage → Ledger → 落盘
⑥
上下文压缩 / 退出 / Thread 销毁前 flush_before_loss() → 抢救性复盘（至少 6 轮后才触发）→ 把还没沉淀的经验写入磁盘

6主 Agent 的两个工具

memory_curate

{
  "action": "add | replace | remove",
  "target": "memory | user",
  "content": "一条简洁事实",
  "old_text": "替换/删除时匹配的旧文本"
}

写入前走 Ledger；默认 auto 直接落盘。满容量时返回 usage，需先 replace/remove。

skill_manage

{
  "action": "create | patch | edit | delete | write_file | remove_file",
  "name": "skill-name",
  "content": "...",
  "scope": "project | user"
}

create/delete 默认 propose 需用户批准；低风险 patch 可 auto。

✓ 审核通过：工具不绕过 Ledger

主 Agent 和后台复盘都通过 ledger.submit() 写入，审计表 evolution_events 有完整记录。

7后台复盘：什么时候触发？

不是每轮都复盘。需同时满足基础门控 + 调度器或信号：

基础门控（与 Memory capture 共用）

  • 本轮必须成功（success=true）
  • 用户消息 ≥ 20 字符，且不是纯「好的/继续/ok」
  • 不是 slash 命令（以 / 开头）

触发条件（满足其一即可）

类型条件复盘内容
定时调度每 N 轮（默认 10）memory 复盘策展记忆
工具轮次累计 tool_rounds ≥ 10（跨多轮累加）技能
高工具密度单轮 tool_rounds ≥ 5记忆 + 技能
用户纠错消息含「不对/应该/别用/error」等记忆 + 技能
明确要求记住流程「记住流程/保存为 skill」等技能
失败后恢复工具报错后又成功执行记忆
load_skill 失败加载技能后工具仍失败技能修补

复盘怎么跑？

启动独立子 Agent（最多 8 步），只给 memory_curate 和 skill_manage 两个工具，读最近最多 8 轮、每条约 1200 字符的 transcript。不修改主对话的 session_messages——这是架构红线，代码已遵守。

主 Agent 本轮调了 memory_curate 会重置 memory 调度计数器，避免刚手动整理完又立刻后台复盘。

8丢失前抢救（Flush）

对话太长要压缩、用户退出、Workbench 销毁 thread 时，上下文里的细节会丢。Flush 在「丢之前」做最后一次抢救性复盘。

  • 触发：post_turn.flush_before_loss()，flush_mode=true
  • 门槛：至少 flush_min_user_turns 轮（默认 6）才执行
  • 超时：单 pipeline 最多 30 秒（flush_timeout_s），超时跳过不阻塞 UI
  • Prompt 用 FLUSH_USER：「会话即将结束，把剩余高价值经验写入策展记忆和技能」

Memory pipeline 同时会 flush_session，Smart Memory 也会做最后一次 capture。

9审批账本（Ledger）

每次变更都是一条 ExperienceMutation，Ledger 按策略决定命运：

auto — 直接落盘 propose — 等用户批准 deny — 拒绝
变更类型默认策略
memory_curate（增删改）auto
skill_patch（低风险）auto；mode=auto_patch 时强制 auto
skill_create / delete / editpropose
后台 review（mode=suggest）全部 propose

审计存在 SQLite evolution_events 表；Workbench 通过 /v1/evolution/pending 拉取待审批项，用户点批准/拒绝走 ledger.approve() / reject()。

10如何注入到 Prompt？

层级内容更新时机
Stable策展 MEMORY + USER 快照；EVOLUTION_GUIDANCE；SKILLS_EVOLUTION_GUIDANCE会话创建时 frozen，中途不变
Volatile<session-evolution> 本 session 新创建的技能通知skill 落盘后追加

这样设计是为了：策展记忆写入后不会让当前会话的 system prompt 突然变长或抖动；新技能则通过 volatile 通知让模型知道可以 load。

11用户能看到什么？

EvolutionBubble（Workbench）

当变更被 propose 时，聊天时间线出现紫色卡片：显示摘要、文件路径、批准/拒绝按钮。

Toast 通知（TUI）

当变更 auto 落盘且 notify=true 时，状态栏出现 💾 摘要提示。

后台复盘本身对用户不可见——没有额外消息插入主对话，符合「静默进化」设计。

12配置开关（config.toml）

[post_turn]
enabled = true          # 关掉则整个 Turn 后处理回退旧行为

[evolution]
enabled = false         # 默认关闭，需手动开启
mode = "suggest"        # suggest | auto_patch
review_model = ""       # 空则用主模型
flush_min_user_turns = 6
notify = true

[evolution.schedulers]
memory_nudge_every_n = 10       # 每 N 轮 memory 复盘
skill_nudge_tool_rounds = 10    # 累计工具轮次阈值
min_tool_calls_signal = 5       # 单轮高工具密度信号

[evolution.ledger]
memory_curate = "auto"
skill_patch = "auto"
skill_create = "propose"
配置组合效果
evolution.enabled=false只有 Smart Memory（若开启），无进化
post_turn.enabled=false回退现网：无 capture 也无 evolution
memory.smart=false + evolution=true进化可独立运行，不依赖 L0-L3

13代码审核结论

✓ 架构合规（对照 EXPERIENCE_EVOLUTION_IMPLEMENTATION.md）

  • 单一 PostTurnOrchestrator 入口，Engine 不直调 capture
  • 共享 gates / scheduler / subagent_runner，无重复实现
  • 工具经 Ledger 写入，无绕过路径
  • Review 用独立 context，不污染 session_messages
  • 策展 stable 快照 session 级 frozen
  • Flush 链覆盖 compact / thread 销毁 / TUI exit

⚠ 使用时需注意

  • 默认关闭：evolution.enabled=false，需主动开启才生效。
  • Stable 快照延迟：本轮写入的策展记忆，当前 session 的 stable prompt 看不到；要新开会话才注入。Volatile 只通知新技能，不通知 memory 变更。
  • 后台复盘消耗 token：触发 review 会额外调一次 LLM（review_model），有成本。
  • 与 Goal 独立：Goal 管目标推进和 token 预算；Evolution 管经验沉淀，两者 token 各自计量、互不干扰。
  • 审批 UI 仅 Workbench：TUI 对 propose 变更暂无 EvolutionBubble，需 Workbench 或 API 批准。
