Workflow 工具 — 实现逻辑完整图解

主 Agent 如何用一次工具调用，编排多个子 Agent 并行/串行协作

核心比喻 一句话定案 三层架构 完整执行流 Workflow IR 四种 Step 运行时细节 子 Agent 启动 策略与容错 进度与 UI 端到端示例 文件索引

🏗️ 用「工地项目经理」理解 Workflow

想象你是一个项目经理（主 Agent / Engine），面前有一栋大楼要盖。

以前的做法：你亲自跑去找每个工人（agent_spawn），等他们干完（agent_wait），再去找下一个——来回跑几十趟，还容易漏人、乱序。

Workflow 的做法：你一次性交出一份施工图纸（Workflow IR JSON），上面写着： 「先勘探地基 → 然后同时铺三栋楼的水电/泥瓦/木工 → 每栋楼先扫再验 → 最后总工汇总报告」。

图纸交给调度中心（WorkflowTool + run_workflow），它按图纸确定性地： 派工人（spawn 子 Agent）、等完工（wait）、把结果传给下一步、实时广播进度。 你只需要在开工前点一次「批准」，不用管中间细节。

0 一句话定案

workflow 是一个普通 ToolSpec：主模型提交 Workflow Spec（JSON IR） → 校验 → 解释执行 → 每步经 DeepSeekAgentRunner 复用 SubAgentManager + run_subagent_loop → 流式 WorkflowProgressEvent + 最终 ToolResult 回主 Engine。

💡 核心执行层是 Workflow IR，不是 Python exec、不是 Node VM。 全程在同一次工具调用内 await 完成，不走 subagent handoff 续回合。

❌ 手工编排（旧方式）

  • 主模型多次调用 agent_spawn
  • 再多次 agent_wait 轮询
  • 合成靠自然语言拼 JSON，易失败
  • 无整体进度视图，只有 Mailbox 卡片
  • 编排逻辑靠模型「临场发挥」，不确定

✅ Workflow 工具（新方式）

  • 一次 workflow 调用提交完整 IR
  • 运行时确定性 fan-out / fan-in / pipeline
  • structured_output 子工具 + 早停 → 结构化终态
  • 可校验、可渲染、可取消的进度树
  • 编排逻辑在 IR 里，执行由 Python 保证

1 三层架构

L1 编排语义层 — workflow/runtime.py

解释 WorkflowSpec：按 phase → step 顺序调度，只依赖 WorkflowRunner 协议 + cancel + callbacks。

  • run_workflow() — 主循环
  • gather_step_outputs() — fanout/pipeline 并发收集
  • run_step() — 单步包装 + snapshot 更新
  • render_workflow_text() — 文本进度摘要

L2 计划层 — models.py + validate.py + template.py

JSON IR 校验、依赖检查、模板变量渲染。可选 adapters/pi_js.py 解析 Pi 风格脚本 meta。

  • parse_workflow_spec() — 结构 + id 唯一 + outputs 拓扑依赖
  • render_template() — {{item}} / {{previous}} / {{outputs.*}}
  • make_step_output() — 生成 text + structured + preview（截断）

L3 DeepSeek 适配层 — tools/workflow_tool.py + agent_runner.py

ToolSpec 入口、审批策略、Engine 注入 cancel/emit、SubAgentManager 绑定。

  • WorkflowTool.execute() — 解析 spec → 跑 workflow → 返回 ToolResult
  • DeepSeekAgentRunner.run() — spawn + poll wait + 转 StepOutput

2 完整执行流（逐步拆解）

Step 0 — 主模型决定使用 workflow

用户明确要求多 Agent 编排 / fan-out / 分阶段审查时，主模型调用 workflow 工具（而非手写多个 agent_spawn）。工具描述和 prompt_guidelines.md 引导模型行为。

Step 1 — Engine 注入上下文

engine/engine.py 在 _execute_single_tool 中，当 tool_name == "workflow" 时注入：

  • engine_cancel_event — 用户按 Esc 取消时联动
  • workflow_tool_call_id — 进度事件主键
  • workflow_emit — 发射 WorkflowProgressEvent 的回调
  • workflow_status_cb — 发射 StatusEvent 一行摘要

Step 2 — WorkflowTool._resolve_spec()

解析入参：纯 spec 对象，或 script + spec（Pi JS 只解析 export const meta，phases 必须来自 spec）。调用 parse_workflow_spec() 做完整校验，失败抛 ToolError。

Step 3 — 创建 DeepSeekAgentRunner

绑定 SubAgentManager + loop_runtime，parent_depth 继承当前 spawn 深度，register_spawned 记录所有 agent_id（用于超时/取消时批量 cancel）。

Step 4 — run_workflow(spec) 主循环

初始化 WorkflowRunContext（outputs 字典、spawned_agent_ids、synthesis_step_ids）和空 WorkflowSnapshot。按 phase 顺序、step 顺序执行，严格串行推进 phase/step 结构，仅在 fanout/pipeline 内部并发。

Step 5 — 每步执行 + 进度广播

每个 step 调用 run_step()：在 snapshot.agents 追加 WorkflowAgentRun(status=running) → await 实际协程 → 更新 done/error → 调用 on_progress(snapshot) → emit WorkflowProgressEvent。

Step 6 — 计算最终结果

_final_result()：若有 synthesis step，取最后一个 synthesis 的 structured（优先）或 text；否则返回所有 step 的 preview 字典。

Step 7 — 返回 ToolResult

成功时 content = 文本进度 + Result 正文；metadata.workflow 含 snapshot、result、logs、duration_ms、errors。主 Engine 拿到后继续当前回合，不触发 handoff 新回合。

3 Workflow IR（施工图纸格式）

顶层结构

{ "version": 1, "meta": { "name": "repo_review", "description": "多视角审查" }, "policy": { "approval_mode": "trusted_workflow", "on_error": "continue", ... }, "phases": [ { "id": "inspect", "title": "勘探阶段", "steps": [ ... ] }, { "id": "synthesis", "title": "汇总阶段", "steps": [ ... ] } ] }
字段含义默认值 / 约束
meta.name工作流名称，snake_case必填，非空
meta.description人类可读描述必填
policy.approval_mode子 Agent 审批策略trusted_workflow
policy.on_error单步失败行为continue（默认）或 fail_fast
policy.max_agents最多 spawn 子 Agent 数10，范围 1~32
policy.concurrencyfanout/pipeline 并发上限4，≤ max_agents
policy.wall_clock_seconds整段墙钟超时600s，超时 cancel 所有子 Agent
phases[]阶段列表，每阶段含 steps至少 1 个 phase，每 phase 至少 1 step
校验规则（validate.py 做了什么？）
  • version 必须为 1
  • phase.id、step.id 全局唯一
  • fanout/pipeline items 非空，最多 16 项
  • 至少有一个会触发 agent 的 step（防空跑）
  • synthesis 引用的 {{outputs.X}} 必须指向已定义且拓扑在前的 step id
  • agent step 必须有 label + prompt；synthesis 必须有 label + prompt_template
模板变量（template.py 运行时替换）
变量含义截断规则
{{item}}fanout/pipeline 当前 item 字符串无
{{previous}}pipeline 当前 item 上一 stage 的 preview已截断
{{outputs.<step_id>}}该 step 的 preview默认最多 2000 字符
{{outputs.<step_id>.full}}该 step 完整 text硬上限 32KB
{{outputs}}所有已完成 step 的一行 preview 索引fanout 每项最多 800 字符

缺失的 output 会渲染为 (missing output: step_id)，所以 synthesis prompt 要能容忍失败 step。

4 四种 Step 类型（怎么「指挥」多个 Agent）

👤

agent — 单人任务

spawn 1 个子 Agent，同步等待完成。对标 Pi 的 await agent(...)。

ctx.outputs[step.id] = 结果

🔀

fanout — 并行扇出

对 items[] 每项 spawn 一个子 Agent，并行执行（Semaphore 限流）。对标 Pi 的 parallel(() => agent(...))。

每项结果存 outputs[f"{id}:{item}"]，合并 preview 存 outputs[id]

🔗

pipeline — 流水线

对 items[] 每项按 stages[] 串行执行多个子 Agent。上一 stage 的 preview 通过 {{previous}} 传给下一 stage。

不同 item 之间可并行（受 concurrency 限制）

🧩

synthesis — 扇入汇总

用 {{outputs.*}} 模板把前面所有分支结果拼成 prompt，spawn 一个汇总子 Agent。通常配 output_schema 产出结构化 JSON。

最终 workflow result 优先取最后一个 synthesis 的 structured

Phase: inspect
  │
  ├─ [agent] api_review ──────────────────────► 子Agent "api reviewer"
  │                                              ctx.outputs["api_review"]
  │
  └─ [fanout] parallel_checks
       items: ["engine", "tools", "workbench"]
       │
       ├──► 子Agent "inspect engine"  ─┐
       ├──► 子Agent "inspect tools"   ─┤ 并行(max concurrency=4)
       └──► 子Agent "inspect workbench"─┘
              ctx.outputs["parallel_checks:engine"] 等
              ctx.outputs["parallel_checks"] = 合并preview

Phase: synthesis
  │
  └─ [synthesis] final
       prompt = "综合 {{outputs.api_review}} 和 {{outputs.parallel_checks}}..."
       ──────────────────────────────────────► 子Agent "final summary"
                                              output_schema → structured JSON
                                              = workflow 最终 result
fanout 并发收集细节（gather_step_outputs）
  • 为每个 item 创建 asyncio.Task
  • 用 asyncio.wait(FIRST_COMPLETED) 逐个收割结果（非一次性 gather）
  • 每项执行前 check_cancel()，取消时 cancel 所有 pending task
  • 单项异常：log 记录；on_error=continue 时跳过该项；fail_fast 时 cancel pending 并抛 WorkflowFailedError
  • 返回结果按 item 原始 index 排序，保证顺序稳定
pipeline 串行传递细节
  • 每个 item 独立跑 stages 循环：prev = None → stage1 → prev → stage2 → ... → return prev
  • 任一 stage 返回 None（失败）→ 该 item 整条 pipeline 失败
  • 不同 item 之间用 Semaphore(policy.concurrency) 并行
  • step_id 格式：{pipeline_id}:{item}（与 fanout 相同模式）

5 运行时核心数据结构

结构作用关键字段
WorkflowRunContext 执行过程中的可变状态 outputs: dict[str, StepOutput] — 每步产出
spawned_agent_ids: list[str] — 所有已 spawn 的 id（取消用）
synthesis_step_ids: list[str] — 记录 synthesis 顺序
StepOutput 单步产出 text — 子 Agent 完整文本
structured — JSON Schema 结构化结果（若有）
preview — 截断后单行摘要（供模板注入）
WorkflowAgentRun 单个 Agent 运行记录（UI 树节点） step_id, label, phase_id
status: queued → running → done / error / skipped
agent_id: spawn 后回填真实 SubAgent id
result_preview, error
WorkflowSnapshot 整体进度快照（推送给 UI） phases[], current_phase, agents[]
agent_count / running_count / done_count / error_count
logs[], duration_ms, result

run_agent_cfg() — 所有 step 类型的公共 spawn 逻辑

  1. 检查 max_agents 上限，超限抛 WorkflowFailedError
  2. 渲染 label（固定 label 或 label_template + {{item}}）
  3. 渲染 prompt（固定 prompt 或 prompt_template + 模板变量）
  4. 调用 runner.run(...) → DeepSeekAgentRunner
  5. 通过 on_agent_id 回调把 agent_id 写回 snapshot 中对应 WorkflowAgentRun
  6. 成功则返回 StepOutput；失败返回 None（由 on_error 策略决定后续）

6 DeepSeekAgentRunner — 怎么真正「派工人」

spawn

1. 构建 SpawnRequest

根据 policy 决定 auto_approve 和 allowed_tools，创建 SpawnRequest 含 prompt、agent_type、model、output_schema 等。

workflow/agent_runner.py → manager.spawn()
↓
SubAgentManager

2. SubAgentManager.spawn()

创建 SubAgent 实例，auto_approve 通过 replace(loop_runtime, auto_approve=...) 按 spawn 注入（非 metadata 写法）。启动 run_subagent_loop 异步任务。

tools/subagent/manager.py
↓
subagent loop

3. run_subagent_loop 独立执行

子 Agent 有自己的工具集（build_subagent_registry + 可选 StructuredOutputTool）。若配了 output_schema，子 Agent 调用 structured_output 成功后早停 break，structured 存入 AgentRunOutput。

tools/subagent/manager.py: run_subagent_loop
↓
wait

4. 轮询等待完成

DeepSeekAgentRunner 每 100ms 调用 manager.get_result(agent_id)，直到 status 非 RUNNING。单 Agent 等待超时 3600s（WAIT_TIMEOUT_MS）。期间持续检查 cancel_event。

↓
output

5. 转换为 StepOutput

FAILED/CANCELLED/INTERRUPTED → 返回 None；成功 → make_step_output(text, structured) 生成 preview。

IR 字段映射到 SubAgent
label / label_templatenickname + assignment.role
agent_typeSubAgentType.parse()（general / explore / review 等）
modelSpawnRequest.model
allowed_toolsSpawnRequest.allowed_tools；analysis_only 时被强制覆盖为只读集
output_schemaSpawnRequest.output_schema → StructuredOutputTool → structured 早停
⚠️ 重要：workflow 内部不再调用 delegate_to_agent / agent_spawn 工具。 它直接走 SubAgentManager API，在同一进程内跑子 loop，保留 Mailbox / 审批 / 取消链路。

7 策略、容错、取消、超时

审批三档（policy.approval_mode）

模式workflow 工具本身子 Agent auto_approve子 Agent 工具集
trusted_workflow（默认） 需用户批准一次（整段编排） True — 子 Agent 内工具自动批准 IR 指定，未指定则用 agent 类型默认集
analysis_only 需用户批准一次 True 强制只读 allowlist（read_file, grep, git_status 等 16 个工具），忽略 IR 写工具
strict 需用户批准一次 False — 写/执行工具在子 loop 返回 Error IR 指定

错误处理（policy.on_error）

模式单步失败时整段 workflow
continue（默认） 返回 None，记 log，snapshot 标 error，继续下一步 最终 success=True，errors 列表记录失败 step
fail_fast 立即 cancel 所有 pending 子任务，抛 WorkflowFailedError success=False，cancel 所有 spawned_agent_ids

取消与超时机制

触发行为
用户 Esc / 父回合取消 engine_cancel_event.set() → 各 step check_cancel → 抛 WorkflowAbortedError → cancel 所有 spawned_agent_ids → snapshot running→skipped
wall_clock_seconds 超时 asyncio.wait_for 包裹整段 _run() → cancel_event.set() → 逐个 manager.cancel(agent_id) → 返回 timed_out
单 Agent 等待超时（3600s） DeepSeekAgentRunner cancel 该 agent，返回 None

核心要点总结

  • Workflow 不是新 Agent 系统，是普通 ToolSpec，在一次工具调用内完成全部编排
  • IR 是核心：主模型写 JSON 计划，Python 运行时确定性执行，不靠模型临场协调
  • 四种 step覆盖所有多 Agent 模式：单人(agent)、并行(fanout)、流水线(pipeline)、汇总(synthesis)
  • 子 Agent 走 SubAgentManager：同进程 run_subagent_loop，spawn + wait，非子进程、非 handoff
  • 结构化终态靠 synthesis/agent 的 output_schema + StructuredOutputTool 早停，非解析 markdown JSON
  • 进度可观测：WorkflowSnapshot 实时推送，Workbench 显示 phase/agent 树
  • 容错可控：continue 跳过失败步 / fail_fast 立即中止；三级审批；墙钟超时 + Esc 取消
  • 模板防爆炸：outputs 默认注入 preview（2000 字符），非全文，避免 synthesis context 超限
