DeepSeek TUI Workflow 全流程图解

用「餐厅」的比喻，带你彻底理解这个系统是怎么工作的

🍽️ 一句话理解整个系统

把这个系统想象成一家高级智能餐厅：

你（用户） = 顾客，坐下来点菜
Engine（引擎） = 大堂经理，接单、协调、确保一切顺利
TurnLoop（轮次循环） = 主厨，真正做菜（调用AI生成回复）
Tools（工具） = 厨房里的各种设备（烤箱、搅拌机...），帮主厨完成特定任务
Approval Gate（审批门） = 食品安全检查员，危险操作必须先过他
PostTurn（后置流水线） = 服务员送完菜后的清理+记录工作
Evolution（进化） = 餐厅的学习系统，记住你的口味偏好

1 一次对话的完整旅程

从你敲下回车，到看到AI回复，中间发生了什么？

📝 第一步：你发送消息

你在界面输入文字，点发送。这时候系统创建一个 SendMessageOp（发送消息操作），放进队列。

💡 比喻：你举手叫服务员，把菜单递过去
↓

🎯 第二步：Engine 接单处理

Engine（大堂经理）从队列里取出你的请求，做准备工作：
① 检查沙箱安全策略
② 回忆历史记忆（你之前说过什么）
③ 组装上下文（系统提示词 + 历史消息 + 工具列表）

💡 比喻：经理看了一眼你的VIP记录，知道你不吃辣，然后把需求转达给厨房
↓

🔥 第三步：TurnLoop 开始流式生成

主厨开始干活了！系统通过 流式（Streaming） 方式调用 DeepSeek API：
- 一个字一个字地返回（你能看到文字逐渐出现）
- 同时监听：AI是不是想要「调用工具」（比如读文件、执行命令）
- 如果超时90秒没响应，自动重试

💡 比喻：主厨一边做菜一边上菜，你能看到盘子里的菜越来越多，而不是等全做完才端出来
↓

🔧 第四步：工具执行（如果需要的话）

如果AI决定需要执行某些操作（读文件、搜索代码、运行命令）：
① 检查权限：这个操作安不安全？需不需要你批准？
② 执行工具：真正去做那个操作
③ 收集结果：把执行结果交回给AI
④ 继续生成：AI拿到结果后继续思考和回复

💡 比喻：主厨做到一半发现需要用烤箱（工具），先问安全员"这个温度行不行"，确认后再烤
↓

✅ 第五步：轮次完成

AI的回复生成完毕，系统发出 TurnCompleteEvent，同时：
- 保存对话历史到数据库
- 记录本次消耗（token数、费用）
- 通知前端"好了，这轮结束了"

💡 比喻：菜上齐了，服务员说"您慢用"
↓

🧠 第六步：后台学习（Post-Turn）

你已经看到回复了，但后台还在默默工作：
- 记忆系统：把这次对话的关键信息提炼存储
- 进化系统：分析"我这次回答好不好？能不能学到什么？"
这些都是异步的，不会让你等待。

💡 比喻：餐后，厨师复盘"这道菜客人满意吗？下次怎么改进？"，同时记录你喜欢的口味

3 核心机制详解（点击展开）

🔄 多轮对话是怎么实现的？

系统用 "round"（回合） 的概念。一次用户输入可能触发多个回合：

  • Round 0：AI回复 + 调用工具A
  • Round 1：拿到工具A结果 → AI继续回复 + 调用工具B
  • Round 2：拿到工具B结果 → AI给出最终回复

最多循环100轮（防止死循环）。每轮都是：流式生成 → 发现工具调用 → 执行 → 把结果塞回去 → 继续生成。

⚡ 流式输出（Streaming）是怎么做的？

核心链路：

  • DeepSeek API 一小块一小块地返回文字
  • TurnLoop 接收每个 chunk，立刻 emit 一个 TextDeltaEvent
  • Handle 把事件放进 event_queue
  • AppServer 通过 SSE 推给前端
  • 前端 实时渲染文字

所以你看到文字是"一个字一个字蹦出来的"——这就是流式。

🛡️ 审批机制（Approval Gate）怎么工作？

不是所有工具都能随便执行。系统有三层判断：

  • 直接通过：只读操作（读文件、搜索）→ 自动批准
  • 缓存命中：最近刚批准过同样参数 → 自动批准
  • 需要确认：危险操作（写文件、执行命令）→ 弹窗问你"可以吗？"

你点"批准"后，结果会被缓存，下次同样操作就不再问了。

🧠 记忆系统（Memory）有几层？

四层分级记忆，像人的大脑：

  • L0 - 原始记录：每轮对话的完整内容（相当于"发生了什么"）
  • L1 - 事实提取：从对话中提取关键事实（"用户喜欢Python"）
  • L2 - 场景理解：理解项目上下文（"这是个电商项目"）
  • L3 - 用户画像：长期偏好（"用户是高级工程师，喜欢简洁风格"）
📦 上下文压缩（Compaction）是怎么回事？

AI的上下文窗口有限（比如128K tokens）。对话太长就装不下了。系统用三种策略：

  • Seam Manager（缝合管理）：对老消息做摘要，保留关键信息
  • Cycle Manager（周期管理）：对话超长时，归档旧部分，只保留简报
  • Emergency Compact（紧急压缩）：消息超500条时强制压缩

这样即使聊了很久，系统也不会"失忆"——它用摘要代替原文。

👶 子代理（Sub-Agent）是什么？

有时候一个任务太复杂，主 Agent 会"派小弟"去做：

  • 主 Agent 创建一个 Sub-Agent（子代理）
  • 子代理独立运行，有自己的工具和上下文
  • 通过 Mailbox（邮箱）向主 Agent 汇报进度
  • 完成后，主 Agent 收到完整结果，继续下一步

多个子代理可以并行运行，效率翻倍。

🔀 Steer（转向）机制是什么？

AI在生成回复的过程中，你可以插一嘴！比如AI正在写代码，你突然说"别用这个库"。

  • 你的消息进入 steer_queue（转向队列）
  • 下一轮开始时，系统会把你的指令注入进去
  • AI看到后会调整方向

不会开新对话，而是在同一轮里修正方向。

🌀 Evolution（进化）系统做什么？

每次对话结束后，进化系统分析：

  • 这次回答质量怎样？有没有被用户纠正？
  • 有没有值得记住的"技巧"（比如某个项目的特殊写法）？
  • 发现信号后，提出建议："要不要把这个记住？"

这样系统会越用越聪明，越来越懂你。

总结：7个关键设计理念

  • 异步优先 — 整个系统跑在 asyncio 上，不会因为等待而卡死
  • 事件驱动 — 所有状态变化都通过事件广播，前端实时感知
  • 流式为王 — 用户体验第一，文字生成一个出一个
  • 分层记忆 — 短期记忆 + 长期画像，越用越懂你
  • 安全门控 — 危险操作必须经过审批，不会偷偷搞事
  • 自我进化 — 每次对话后自动复盘，提炼可复用技能
  • 弹性压缩 — 对话再长也不怕，多种策略保证不超限
