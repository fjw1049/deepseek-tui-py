## 一、审核摘要（先读这一段）

### 1.1 审核结论（一句话）

**仓库存在大量"写完没接通"的代码，主要不是"曾经在用、被替代后没删"，而是"从加进来的那一刻起就没接进系统"。** 三个 mega commit（Stage 0 baseline `b9ab4c9`、Stage 4.6-4.9 `20029b9`、Stage 6.next `502ef70`）是孤岛的主要源头。

### 1.2 几个数字

| 指标 | 数量 |
|---|---|
| 审核覆盖的 src/ 文件总数 | 122 个 .py 文件（不含 `__pycache__`） |
| 直接被路由入口（CLI/HTTP）触达的核心 .py 文件 | ~85 个 |
| 需要调整的文件（含确认的 bug + 设计偏差） | **37 项**（详见第六部分） |
| 完全 0 实例化 / 0 调用的孤岛文件 | **19 项**（详见第七部分） |
| 孤岛文件总行数 | ~3,200 行（占 src/ 总行数 约 12%） |
| HANDOVER 第 9 节已登记的孤岛 | 14 项（占 19 项中的 74%） |
| HANDOVER 没登记的孤岛 | 5 项（占 19 项中的 26%） |
| HANDOVER 中发现的事实性误诊 | 1 处（详见第九部分） |

### 1.3 影响最严重的 4 处事实

按"如果用户今天上手会发生什么"排序：

1. **HTTP `/threads/*` 11 条 Rust-parity 路由全部空转**：`app_server/routes.py:83-217` 11 条路由全部依赖 `request.app.state.thread_manager`，但全仓**没有任何 commit** 设过这个属性。任何人发 `POST /threads`、`GET /threads`、`/threads/{id}/turns` 等 → 全部返回 `{"ok": false, "error": "runtime thread manager not configured"}`。从 commit `20029b9`（2026-05-09）起至今 30 多个 commit 这条路一直是死的。
2. **`tools/knowledge_tools.py::ReviewTool` 调用即崩**：`line 184-204` 同时 import 两个**不存在**的模块（`deepseek_tui.client.models`、`deepseek_tui.client.events`）。但 `ReviewTool` 注册在 `builder.py:179`。LLM 一旦调用 `review` 工具，立即 `ModuleNotFoundError`。
3. **`app_server/thread_manager.py:552` 同样的 import 死路径**：`from deepseek_tui.client.deepseek_client import DeepSeekClient` —— 模块名错（应为 `deepseek_tui.client.deepseek`）。今天因 RuntimeThreadManager 整体不接通而被掩盖；任何接通尝试会立即触发 `ModuleNotFoundError`。
4. **3 个不同的 `ThreadRecord` 类共存**：`protocol/threads.py::Thread`、`app_server/runtime.py::ThreadRecord`、`app_server/runtime_threads.py::ThreadRecord`。`app_server/__init__.py` 同时 export 后两个，名字相同 → `runtime.py` 那个胜出 → `runtime_threads.py::ThreadRecord` 静默被遮蔽。任何 `from deepseek_tui.app_server import ThreadRecord` 拿到的不是写代码的人以为的那个。

### 1.4 推荐的清理优先级（不在本文档执行）

| 优先级 | 工作量 | 影响 |
|---|---|---|
| **P0 — 修硬 bug** | ~30 行改动 | 修两处 `ModuleNotFoundError`（`thread_manager.py:552`、`knowledge_tools.py:184/190/200`） |
| **P1 — 物理删除明确死代码 (Group A)** | 删 ~800 行 + 12 个 `__init__.py` export | 减少 12% 的"看起来在用其实没用"代码 |
| **P2 — 二选一决策 (Group B)** | 接通 or 删，每项都是单独 commit | 处理 3 处大块孤岛：RuntimeThreadManager / RlmTool / protocol Pydantic 校验 |
| **P3 — 给 R3 类孤岛加 marker** | ~10 行注释 | 防止下个接手 AI 浪费时间 grep |

## 二、审核范围与方法论

### 2.1 起点

按用户要求，**从路由的入口文件开始**。仓库存在两个路由入口：

```
src/deepseek_tui/__main__.py
└── from .cli.app import app
    └── src/deepseek_tui/cli/app.py        ← CLI 路由（22 个 typer 子命令）
        └── serve / app-server 子命令
            └── from deepseek_tui.app_server import run_http
                └── src/deepseek_tui/app_server/server.py
                    └── build_fastapi_app
                        └── src/deepseek_tui/app_server/routes.py  ← HTTP 路由
```

从这两个入口开始，递归 grep 引用关系，覆盖到所有"被路由触达的核心代码"，再对比 src/ 全体 .py 文件，找出"没人触达的孤岛"。

### 2.2 工具

| 用途 | 工具 |
|---|---|
| 看文件内容 | `Read` |
| 找符号引用 | `Grep`（ripgrep 包装），`output_mode: files_with_matches` 找文件、`content` 看上下文 |
| 找文件创建/修改时间 | `git log --diff-filter=A -- <file>`（创建）、`git log --oneline -p -- <file>`（演化）|
| 找符号曾被谁用过 | `git log --all -p -S "<symbol>" -- '*.py'`（pickaxe）|
| 看具体 commit 改了什么 | `git show <hash> --stat`、`git show <hash> -- <file>` |

### 2.3 输出口径

每条孤岛/调整项都给出：

- **文件路径 + 行号**（精确到段落）
- **首次进 git 的 commit hash + message 摘要**
- **失效原因**（用 R1-R5 分类）
- **替代代码位置**（如果有）
- **HANDOVER 是否已登记**
- **推荐处理方式**

| 分类 | 含义 | 典型案例 | 占比 |

### 3.1 Stage 0 baseline 的来历（解释为什么 R1 占比这么高）

`b9ab4c9 Stage 0: baseline commit + engineering environment calibration`（2026-05-06，第 1 个 commit）的 commit message 自己写：

> "an old `ARCHITECTURE_AUDIT.md` / `COMPLETION_SUMMARY.md` '95% / production-ready' claim. **Real parity is ~5%**."

也就是说：**仓库的 Stage 0 baseline 是把上一代 AI 写的 Python 代码（自称 95% 完成、实际 ~5% parity 的占位代码）整体导入 git**。Stage 0 自己没动这些代码，只是建立 ruff/mypy/test 框架并校准 `5%` 的事实。

后续从 Stage 1 开始，每个 stage 的 AI 都是**绕过老代码、写新版本**，但**几乎从不删老文件**。最终老/新代码并存。下面每条 R1 都是这个模式的产物。

#### 5.1.2 三个 `ThreadRecord` 类同名冲突详细分析

self.event_bus: AsyncBroadcast[RuntimeEventRecord] = AsyncBroadcast(
