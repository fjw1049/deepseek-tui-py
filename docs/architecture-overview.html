核心思想 · 一句话看懂这套系统

整个系统是一台 「会用工具的 LLM 状态机」：用户的一句话进来，引擎反复地 「问模型 → 模型要调工具 → 执行工具(过审批和沙箱) → 把结果喂回模型」， 直到模型不再要工具为止 —— 这个 工具往返循环 (tool round-trip loop) 就是大脑， 其它一切（安全、记忆、上下文压缩、MCP、子代理、双前端）都是围绕它的「外挂器官」。
① 解耦：actor 邮箱模型
EngineHandle 是引擎和前端之间唯一的信箱：前端投「op」进去，引擎吐「event」出来。所以同一个引擎既能驱动终端 TUI，也能驱动 HTTP+SSE 的桌面 GUI，互不感知。
② 安全：双闸门
每个有副作用的工具调用都过两道门——先审批门(要不要问用户)，再OS 沙箱(macOS Seatbelt 限制能写哪、能不能上网)。审批结果按「语义指纹」缓存，批准 git status 绝不等于批准 git push。
③ 省钱省 token：什么都「惰性 + 分层」
工具默认不全发给模型(惰性加载)、超大工具输出落盘(溢出)、长对话自动摘要(压缩)、记忆按 L0→L3 逐层蒸馏。处处为「有限上下文窗口 + KV 前缀缓存」做精打细算。
设计基因：忠实移植 Rust
几乎每个模块都标注 mirrors crates/tui/src/...rs。数据结构、路径布局、加载链、沙箱 profile 都按 Rust 原版 1:1 复刻 —— 这是理解任何分歧时的「权威源」。
设计哲学：层层兜底，绝不崩
模型吐了畸形 JSON？修复阶梯。吐了假工具包装？流式过滤器。上下文爆了？三级恢复+压缩+周期归档。工具输出太大？落盘+按需回取。每条出错路径都有 recovery，宁可降级不崩溃。

💡 为什么是「循环」而不是「一问一答」？ 因为 Agent 要自主完成任务：读文件→想→改代码→跑测试→看结果→再改……每一轮模型自己决定下一步调什么工具，引擎只负责忠实地执行+把结果还给它，并在循环里见缝插针地做安全、压缩、诊断。

② 桌面 GUI（Electron）

app_server/ · FastAPI · packages/workbench

serve --http 起 Runtime API（默认 :7878）。把「单用户终端引擎」包装成多会话 HTTP + SSE 后端给 Electron 用。

  • /v1 路由：threads / turns / events(SSE) / approvals / elevations / jobs / mcp / skills / tasks / automations / workspace
  • RuntimeThreadManager 按 thread 持有 Engine，LRU 驱逐
  • 审批/提权桥：用 asyncio.Future 把工具调用挂起，GUI 决策后 resolve 引擎才继续

③ 一次性 / 服务化

无界面用法，复用同一引擎与工具栈。

  • -p "..." / exec：单次问答，直接驱动 Engine 打印流
  • review：对 git diff 做 AI 代码评审
  • serve --stdio：JSON-RPC 管道模式
  • mcp-server：把本程序自己当成 MCP 服务暴露给别的 Agent（反向集成）

B引擎大脑 —— 编排者与它的接缝

engine/（~8.5K 行）。Engine 是总指挥；EngineHandle 是信箱；TurnLoop 是流式驱动器；ToolRuntime 是装配车间。

Engine · 总编排

engine/engine.py（2.5K 行核心）
核心思想：把「一轮对话」分解成可被横切关注点(安全/记忆/压缩/诊断)插入的状态机，每个关注点都是循环里的一个 checkpoint。
  • run()：op 主循环，串行处理 turn，支持取消
  • 工具分发：特殊内建(tool_search/code_execution/parallel/user_input) → 外部 MCP → 注册表工具
  • /undo 快照：写文件类工具执行前快照原文件
  • 子代理 handoff：直接子代理完成 → 注入 <subagent.done> 续跑本轮
  • 成本累计：从 usage 算 USD/CNY + 缓存命中累计

EngineHandle · actor 信箱

engine/handle.py
核心思想：引擎不认识前端。一切交互降维成两个队列 + 几个事件，前端可任意替换。
  • op_queue 入：SendMessage / Cancel
  • event_queue 出：文本/思考/工具/审批/状态…
  • steer_queue：中途插话；cancel_event：打断
  • pending_user_inputs：工具向用户问问题的 Future
  • 事件桥：EngineEvent → HookEvent，喂给 Hooks 系统

TurnLoop · 流式驱动器

engine/turn_loop.py
核心思想：把不可靠的网络流变成可靠的一轮结果——三级重试 + 三道流守卫 + 文本兜底解析。
  • SSE 增量：文本 / 思考(reasoning) / 工具调用
  • 重试三级：透明重试 / 断流重试 / 上下文恢复
  • 守卫：90s 空闲 · 30min 墙钟 · 10MB 内容上限
  • 只发激活工具：每步按 active 集重建请求(惰性加载)
  • 兜底：模型用文本写工具调用也能解析出来

ToolRuntime · 装配车间

tools/runtime.py
核心思想：唯一装配点。所有 manager 与 context 在这里一次性配齐、按 feature flag 开关，避免「各工具各自初始化」的碎片化。
  • 组装：Registry + TaskMgr + SubAgentMgr + Mailbox + McpMgr + LspMgr + AutomationMgr
  • 挂 Policy + 网络策略 + 沙箱策略 + 工作目录
  • 有 API key → 真实执行器，否则 stub（测试友好）
  • 提供 shutdown() 优雅清场

C工具层 —— Agent 的「手脚」（最大子系统 ~14.7K 行）

🧩 亮点工具：RLM（递归语言模型 REPL）

tools/rlm/ · repl.py · turn.py

给模型一个受限 Python REPL：模型写代码块 → 在沙箱命名空间执行(禁 eval/exec/compile) → 注入 llm_query()/rlm_query()/FINAL() 等助手 → 把执行结果反馈给模型，最多 25 轮迭代。

用途：处理超长上下文 / 数据密集任务时，让模型「边写代码边算」而不是把所有数据塞进 prompt。

💾 亮点机制：溢出 (Spillover)

tools/spillover.py · retrieve_tool_result.py

工具输出 >阈值时写到 ~/.deepseek/tool_outputs/<id>.txt，只把头部(~32KB)+「用 retrieve 取更多」提示回给模型。

模型可用 retrieve_tool_result 以 summary/head/tail/lines/regex query 五种模式按需取，无需重跑工具，也不撑爆上下文与 KV 缓存。7 天自动清理。

指纹缓存 + 网络 + 密钥

approval_cache · network/policy · secrets/
批准 git status ≠ 批准 git push：按语义指纹缓存，不按工具名。
  • 网络：域名 allow/deny，deny 优先，子域通配
  • 密钥：运行时只走 env/config（避开 keychain 弹窗）；文件存储强制 0600
  • 策略可热追加规则(带文件锁)

E上下文治理 —— 在有限窗口里塞下无限对话

引擎支撑模块的精华。一整套「分层防线」管理 token 占用，让对话能无限长地跑在固定大小的 LLM 窗口里。

惰性工具加载
tool_catalog.py：启动只激活 ~30 个常用工具 + tool_search；其余「延迟」。模型用 tool_search_bm25/regex 搜出来或点名才激活 → 初始 prompt 大幅瘦身。
容量控制器
capacity.py：按 token 占用率分 低/中/高 风险带，触发「定向刷新 / 工具重放 / 重新规划」等守卫动作（默认关）。
消息压缩
compaction.py：超阈值时调 LLM 把旧消息摘要成 <archived_context>，保留最近 4 条 + working-set 引用的消息。500K 硬地板防止压坏 KV 缓存。
周期归档
cycle/seam_manager.py：对话破 768K token 时，整段归档到磁盘，生成 ~3K 的「交接简报」开新窗口续跑（默认关，opt-in）。

F1分层记忆 —— L0→L3 逐层蒸馏（~2.2K 行）

memory/。核心思想：把昂贵的 LLM 抽取推迟到空闲时做，原始对话即时落盘，慢慢蒸馏成结构化知识；召回用混合检索预先索引好。

L0 原始层
l0_recorder/search：每轮对话即时写 per-thread JSONL；conversation_search 用 BM25 检索原始档案。「上周聊过啥」走这层。
L1 事实层
l1_extractor：空闲/满 N 轮时，调 LLM 把对话抽成结构化记忆（persona/episodic/instruction）+ 置信度，去重后入库。
L2 场景层
l2_scenes：把 L1 按「场景」聚类成 markdown 块，作为导航上下文注入（「最近在调认证 bug」）。
L3 人设层
l3_persona：聚合所有 persona 类记忆成 persona.md，以 <persona> 块注入（「用户偏好 async/await」）。

存储 + 混合检索

native/store.py · embedding · hybrid_search

SQLite + FTS5 全文(jieba 支持中文)+ 可选 向量嵌入，用 RRF 倒数排名融合 BM25 与语义分。带时间衰减 + 当前工作区加权。嵌入不可用时优雅降级到纯关键词。

协调器 + 双模式

coordinator.py · gates.py · user_memory.py

recall(回合开始,带 500ms 超时)注入 L1/L2/L3；capture(回合后,异步)落盘+触发抽取，质量门跳过确认类/过短回合。模式：auto(智能) / hybrid(智能+memory.md) / manual(纯文件)。

G扩展层 —— MCP / LSP / Hooks / Skills

四个「插拔口」，让系统能接外部工具、拿编译诊断、被观测、加技能。

🔌 MCP

mcp/manager · transport · execute · preload
把外部 MCP 服务器的工具桥成一等公民 mcp_<server>_<tool>。
  • 传输：stdio 子进程 / SSE-HTTP
  • 冷启动优化：工具列表按配置 hash 落盘缓存，后台 preload 并行发现，绝不阻塞首轮
  • stale-while-revalidate：配置变了先用旧缓存再后台刷新

🩺 LSP

lsp/manager · diagnostics · hooks
编辑文件后自动拿语言服务器诊断，把编译错误注入回对话让模型下一轮看到。
