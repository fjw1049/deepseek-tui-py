Experience Evolution

自主迭代复盘逻辑
通俗图解

主 Agent 忙着回你消息的时候，后台还有一个「复盘员」在默默翻聊天记录， 把值得长期记住的东西写进笔记本，把可复用的操作流程整理成技能。 这就是 Evolution Review（进化复盘）。

一句话比喻

把主 Agent 想象成正在上课的学生，Evolution Review 就是课后悄悄做笔记的助教。 学生不用停下来写日记——助教在后台翻这节课的对话，只记下「下次还能用」的东西： 项目习惯、踩过的坑、标准操作流程。废话、临时进度、一次性指令，一律不写。

01

一轮对话结束后，发生了什么？

用户发完消息、Agent 回复完毕，才算一个 Turn 结束。此时 PostTurn 编排器接手。

1

打包「证据包」TurnEvidence

把这一轮的对话原文、用户说了什么、用了几次工具、是否成功等信息，打成一个小包裹。

同步 · 毫秒级
2

过「质量门槛」

太短的消息（「好的」「继续」）、斜杠命令、失败的回合——直接跳过，不值得复盘。

Gate 过滤
3

判断「该不该复盘」

定时器到了？工具用了很多轮？用户纠正了 Agent？说了「记住这个流程」？满足任一条件就触发。

触发条件
4

后台启动「复盘子 Agent」

异步运行，不阻塞你继续聊天。它读最近最多 8 轮对话，调用 memory_curate / skill_manage 工具写笔记。

后台异步
5

账本审批 & 落盘

Experience Ledger 按策略决定：自动写入、等你批准、还是拒绝。通过后写入 MEMORY.md / USER.md / SKILL.md。

Ledger 审计
02

什么时候会触发复盘？

不是每句话都复盘。系统用「定时提醒 + 异常信号」两种方式决定时机。

⏰

定时提醒（Scheduler）

  • 记忆复盘：每 10 个用户回合提醒一次（可配置 memory_nudge_every_n）
  • 技能复盘：累计工具调用轮数达到 10 轮时提醒（skill_nudge_tool_rounds）
  • 主 Agent 自己调了 memory_curate，记忆定时器会重置——刚写过就不重复催
🚨

异常信号（Signals）

即使定时器没到，出现以下情况也会立刻触发：

工具轮数 ≥ 5（复杂任务） 用户纠正（「不对」「应该用…」） 用户说「记住流程 / 保存为 skill」 工具报错后又成功（踩坑恢复） 加载技能失败（技能缺口）

03

复盘子 Agent 在干什么？

它是一个独立的、受限的 LLM 循环，只有「写笔记」和「管技能」两个工具，碰不到你的文件系统和终端。

SYSTEM PROMPT
EVOLUTION_REVIEW_SYSTEM · 复盘员的「人设」
You are a background memory curator. Review the conversation excerpt and update curated memory and/or skills using only the provided tools. Be conservative: only persist durable, reusable knowledge. Do not repeat volatile task state.
复盘模式 用户提示开头 可用工具
只审记忆 「Review this turn for durable agent notes (MEMORY)…」 memory_curate
只审技能 「Review this turn for repeatable procedures…」 skill_manage
两者都审 「Review this turn for both curated memory and skills.」 两个工具都有
丢失前抢救（Flush） 「Session is ending… Flush any remaining durable notes…」 两个工具都有，更积极

🔒 重要隔离原则

  • 复盘读的是对话副本，不会污染主 Agent 的 session 上下文
  • 最多回看最近 8 轮对话、每条消息截断到 1200 字符
  • 最多走 8 步工具循环（review_max_steps），防止无限折腾
  • 失败了会自动重试 1 次，再失败才放弃并提示
04

写什么？不写什么？

复盘员的核心原则是保守——宁可漏记，也不乱记。

✅ 值得写进长期记忆

  • 项目约定（「这个 repo 用 pnpm」「测试跑 pytest」）
  • 用户偏好（「回复用中文」「不要自动 commit」）
  • 踩过的坑（「别改 git config」「hook 会改文件」）
  • 可复用的操作流程（部署步骤、Code Review 清单）

❌ 不该写

  • 当前任务进度（「正在改第 3 个文件」）
  • 一次性指令（「帮我把这个函数改名」）
  • 临时变量、调试输出、错误堆栈原文
  • 用户随口说的废话和确认语

写到哪里 · 记忆

MEMORY.md / USER.md

target=memory → Agent 自己的笔记
target=user → 用户画像
存在 ~/.deepseek/memories/

写到哪里 · 技能

SKILL.md

完整的操作流程文档，带 YAML 头
默认写在项目 .deepseek/skills/

怎么用上

注入下次对话

记忆 → 会话开始时稳定注入 prompt
技能 → 会话中动态通知「有新技能可用」

05

丢失前抢救（Flush）

关窗口、换会话、上下文被压缩之前——最后一遍扫对话，把还没沉淀的好东西捞回来。

📦

触发时机

退出 TUI、新建会话、上下文 compact 之前，Orchestrator 调用 flush_before_loss。

⏳

最低门槛

至少聊过 6 个用户回合（flush_min_user_turns）才跑，避免刚开聊就抢救。

🔥

同步执行、记忆+技能全开

Flush 用更积极的提示词，memory 和 skill 两条线同时审，超时默认 30s。

同步 · 有超时保护
06

账本：写完就能用吗？

不一定。Experience Ledger 是「审计员」，每条变更都要过策略审批。

变更类型 默认策略 含义
memory_curate（增删改记忆） auto 直接写入，下次对话生效
skill_patch（低风险补丁） auto 小改动直接合并进 SKILL.md
skill_create（新建技能） propose 先提案，等你点「批准」才创建
复盘来源 + mode=suggest propose 后台复盘的结果默认要你确认

三种结局

  • auto → 立刻落盘，TUI 可能弹出「💾 已保存…」
  • propose → 进入待审批列表，你确认后才写入
  • deny → 策略拒绝，记一笔审计日志，不改动任何文件
07

和 Smart Memory、主 Agent 什么关系？

三个系统并行，各管各的，不抢活。

Smart Memory

自动结构化记忆

后台自动提取 L0→L3 层级记忆
管「系统知道什么」
用户不直接编辑

Evolution 复盘

Agent 策展 + 技能演化

复盘员 + 主 Agent 工具双通道
管「下次怎么做更好」
产出 MEMORY / SKILL 文件

主 Agent

实时对话

也可以主动调 memory_curate / skill_manage
调了会重置定时器，避免重复催
专注回你消息，不被复盘阻塞

口诀

Memory 管「知道什么」（自动沉淀的事实） · Evolution 管「下次怎么做更好」（策展笔记 + 操作流程） · 主 Agent 管「现在帮你干活」（实时响应）

08

关键配置速查

在 config.toml 的 [evolution] 和 [evolution.schedulers] 段调整。

配置项 默认值 通俗解释
enabled true 总开关。关了就没有后台复盘。
memory_nudge_every_n 10 每聊 10 轮，提醒复盘员看看要不要更新记忆。
skill_nudge_tool_rounds 10 工具累计跑满 10 轮，提醒复盘员看看要不要写技能。
flush_min_user_turns 6 关会话前抢救的最低回合数。
review_max_steps 8 复盘子 Agent 最多走几步工具调用。
mode suggest suggest=复盘结果要你批；auto_patch=低风险自动写。
源码入口： evolution/pipeline.py（调度）→ evolution/review/runner.py（复盘执行）→ evolution/ledger.py（审批落盘）→ evolution/prompts.py（EVOLUTION_REVIEW_SYSTEM 等人设文案）。 用浏览器打开本文件即可阅读，无需启动服务。
