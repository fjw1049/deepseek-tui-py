DeepSeek TUI — Hooks 实现逻辑

项目里有两条互不替代的 Hook 管道，共用 [hooks] 配置段但读不同字段。 下图按真实代码路径绘制：hooks/build.py、 engine/handle.py、 hooks/executor.py、 engine/engine.py、 tools/shell_tools.py。

管道 A — 可观测性 HookDispatcher 管道 B — 生命周期 HookExecutor 分支 / 条件 入口（TUI / HTTP）

0. 总览：两套 Hook 从哪分开 mental model

管道 A · 可观测性（Observer）

  • 类：HookDispatcher + Sinks
  • 构建：build_hook_dispatcher()
  • 挂载：EngineHandle(hooks=…)
  • 触发：handle.emit(EngineEvent) → 桥接 → dispatcher.emit(HookEvent)
  • 配置：stdout、jsonl_path、webhook_urls、legacy shell_hooks
  • 目的：记日志 / Webhook，不阻塞主流程（失败吞掉）

管道 B · 生命周期（Lifecycle）

  • 类：HookExecutor
  • 构建：build_lifecycle_hook_executor()
  • 挂载：Engine.hook_executor + tool_context.metadata
  • 触发：显式 run_lifecycle_hook(event) 或工具路径内调用
  • 配置：[[hooks.hooks]] 每条 event + command
  • 目的：跑你的 shell 脚本，可带 DEEPSEEK_* 环境变量
易混点：管道 A 的 shell_hooks 是按 HookEvent 类型 过滤后跑命令（旧式观测）； 管道 B 的 [[hooks.hooks]] 是按 session / tool / message 生命周期跑命令。名字都叫 hook，实现完全不同。

用户启动 deepseek-tui 或 deepseek serve --http

Config.hooks（HooksConfig） enabled · default_timeout_secs · hooks[] · jsonl_path · webhook_urls · …
并行构建两条管道

TUI: tui/app.py

EngineHandle(hooks=dispatcher)传给 Engine.create(handle=…)

HTTP: app_server/runtime.py

AppRuntime.hooks同样 build_hook_dispatcher；prompt 路径直接 hooks.emit

Engine.create()

Engine.hook_executorengine/engine.py
tool_context.metadata["hook_executor"]供 shell_tools 读 shell_env

若 handle 尚无 hooks

2. 管道 A 实现逻辑：EngineEvent → JSONL/Webhook HookDispatcher

Engine / turn_loop 产生事件ToolCallEvent · ToolResultEvent · TurnStartedEvent · …
await handle.emit(event)engine/handle.py — 主路径
① event_queue.put（TUI 消费） ② await _bridge_to_hooks_safe
_bridge_to_hooks(event)EngineEvent 类型 → HookEvent 映射表
能映射？需 set_response_id 才有 response_* 事件
await dispatcher.emit(hook_event)hooks/dispatcher.py — 并发 fan-out 到所有 sink
Sinks 写入StdoutHookSink · JsonlHookSink · WebhookHookSink · ShellHookSink

EngineEvent → HookEvent 映射（handle._bridge_to_hooks）

EngineEventHookEvent备注
TurnStartedEventResponseStartEvent需要 response_id
TextDeltaEvent / ThinkingDeltaEventResponseDeltaEvent流式 delta
TurnCompleteEvent / TurnCancelledEventResponseEndEvent
ToolCallEventToolLifecycleEvent phase=start
ToolResultEventToolLifecycleEvent phase=complete|error
SessionStartedEvent / EndedSessionLifecycleEvent
ApprovalRequired/ResolvedApprovalLifecycleEvent
try_emit 为何不走路径 A？

后台 drainer（如流式输出积压消费）调用 handle.try_emit()：只 put_nowait 进 event 队列，故意不调用 _bridge_to_hooks，避免慢 sink / shell 拖死生产者或泄漏子进程。

3. 管道 B 实现逻辑：生命周期脚本 HookExecutor

调用方显式触发TUI / Engine / commands — 见下表「谁调用」
Engine.run_lifecycle_hook(event, …)组装 HookContext
has_hooks_for_event(event)?config.enabled && hooks[] 里有匹配 event
hook_executor.execute(event, ctx)hooks/executor.py
每条 [[hooks.hooks]]：condition 匹配？tool_name / tool_category 等
asyncio.create_subprocess_shell(command)env = 系统环境 + DEEPSEEK_*
success? continue_on_error?失败可 break 后续 hook

谁在哪里调用 run_lifecycle_hook

event调用位置时机
session_starttui/app.pyEngine 启动后
session_endtui/app.py优雅退出
message_submittui/app.py用户回车、发往模型前
mode_changetui/commands/handlers.pyPlan/Agent/YOLO 切换
on_errortui/app.py传输/容量等错误
tool_call_beforeengine/engine.py _execute_single_tool每个工具执行前
tool_call_after同上工具返回后（填充 result/success）
shell_envtools/shell_tools.py _shell_env_from_hooks仅 exec_shell spawn 前

HookContext → 环境变量（节选）

# hooks/executor.py HookContext.to_env_vars()
DEEPSEEK_TOOL_NAME / DEEPSEEK_TOOL_ARGS / DEEPSEEK_TOOL_RESULT
DEEPSEEK_MESSAGE / DEEPSEEK_MODE / DEEPSEEK_WORKSPACE / DEEPSEEK_SESSION_ID
DEEPSEEK_ERROR # on_error

4. 特殊分支：shell_env exec_shell only

模型调用 exec_shell
_shell_env_from_hooks(context, command)从 metadata 取 hook_executor
collect_shell_env_async(HookContext)execute("shell_env") 跑所有匹配 hook
parse_env_lines(stdout)KEY=VALUE 或 export KEY=VALUE 按行解析
子进程 env = os.environ ∪ extrahook 失败 → 不注入，不阻止 shell

与管道 B 其它事件不同：shell_env 的 stdout 不是给人类看的日志，而是被解析进环境变量。

5. 时序：一次工具调用时两条管道并行 sequence

步骤
管道 B（生命周期）
1
_run_lifecycle_hook("tool_call_before", ctx)
2
_execute_single_tool_impl → 若 exec_shell：collect shell_env
3
_run_lifecycle_hook("tool_call_after", ctx + result)
步骤
管道 A（可观测）
a
handle.emit(ToolCallEvent) → ToolLifecycle start → Jsonl
b
（工具执行中…）
c
handle.emit(ToolResultEvent) → ToolLifecycle complete → Jsonl

管道 A 与 B 无直接调用关系；它们由 Engine 在不同代码行各自触发，配置也来自同一 [hooks] 表的不同字段。

6. config.toml 字段 → 代码路径 config map

配置项管道代码
[hooks] enabled = trueBHookExecutor 总开关
[[hooks.hooks]] event + commandBLifecycleHookEntry → execute()
condition = { … }B_matches_condition()
background / continue_on_error / timeout_secsB_execute_sync / _execute_background
jsonl_path = "…"AJsonlHookSink
webhook_urls = […]AWebhookHookSink（重试 2 次）
stdout = trueAStdoutHookSink
[[hooks.shell_hooks]]A（legacy）ShellHookSink；也会 merge 进 B 的 hooks 列表
示例配置（可复制） 
# ~/.deepseek/config.toml
[hooks]
enabled = true
jsonl_path = "~/.deepseek/hooks-events.jsonl"

# 管道 B：生命周期脚本
[[hooks.hooks]]
event = "session_start"
command = "echo started >> /tmp/ds-hook.log"

[[hooks.hooks]]
event = "shell_env"
command = "echo MY_TOKEN=demo"
