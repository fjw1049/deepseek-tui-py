Engine 完整逻辑指南

文件路径：src/deepseek_tui/engine/engine.py
Engine 是整个 DeepSeek TUI 的「大脑」—— 接收用户消息，调用 LLM，执行工具，把结果推回界面。

1. Engine 是什么？

一句话：Engine 是一个异步编排器，它不停地从 EngineHandle 读取「操作指令」，驱动 LLM 多轮对话 + 工具调用，并通过事件队列把进度实时推给 TUI 界面。

你可以把它想象成餐厅里的主厨：

  • EngineHandle = 点单窗口（收用户消息、取消请求、中途加菜）
  • LLMClient = 外聘顾问（DeepSeek 模型，给出回答或「需要用什么工具」）
  • ToolRegistry = 工具箱（读文件、跑 shell、搜代码……）
  • 事件队列 = 出菜口（文字流、工具结果、审批弹窗，推给前端渲染）

Engine 本身不渲染界面，只负责逻辑编排。界面（TUI）通过 handle.events() 订阅事件，通过 handle.send_message() 发消息。

4. 主循环 run()

Engine 启动后进入无限循环，这是整个程序的入口：

flowchart TD Start["run() 启动"] --> Loop{"while True"} Loop --> Wait["await handle.next_op() 阻塞等待"] Wait --> Check{操作类型?} Check -->|"SendMessageOp"| Handle["_handle_send_message(op)"] Check -->|"CancelRequestOp"| Cancel["handle.cancel_event.set()"] Cancel --> Loop Handle --> Loop Loop -->|"asyncio.CancelledError"| Raise["向上抛出，Engine 停止"]

取消机制细节：收到 CancelRequestOp 时，除了记录日志，还会强制设置 cancel_event。这是防御性设计——即使调用方忘了调 handle.cancel()，正在进行的流式请求也会在下一个 chunk 检测到取消并停止。TurnLoop 每个 stream chunk 都会检查这个标志。

Plan 模式特殊逻辑

函数 should_force_update_plan_first(mode, content) 检测：

  • 用户在 plan 模式下说「quick plan」「give me a plan」等
  • 但没有说「explore the repo」「read the code」等探索意图
  • → 自动追加一条 System 消息，要求 AI 先探索代码库再写计划

完成后的收尾

  • 取消时：不更新 session_messages（本轮未完成，历史不变）
  • 成功时：把 working_messages 写入 session_messages
  • 累计 session 级 cache hit/miss（不是单 turn 的 99%，而是整个会话累计）
  • 用 pricing 模块算费用，累加到 session_cost_usd/cny
  • 发 TurnCompleteEvent（含 assistant 消息、usage、费用、cache 统计）
  • 调用 _auto_persist_session 持久化到 current.json

6. 多轮对话循环 _run_conversation

这是 Engine 的心脏—— Agent 循环：问 LLM → 调工具 → 再问 LLM，直到模型说「好了，不用工具了」。

flowchart TD Start["开始 _run_conversation"] --> Inc["turn_counter += 1"] Inc --> Cycle{cycle 启用?} Cycle -->|是| MC["_maybe_advance_cycle"] Cycle -->|否| Round MC --> Round subgraph Round["for round_idx in 0..max_tool_round_trips+1"] R1["① drain_steers 注入中途消息"] R2["② run_pre_request_checkpoint 容量预检"] R3{"③ 消息>500 或 should_compact?"} R3 -->|是| C["compact_messages_safe 压缩"] R3 -->|否| R4 C --> R4["④ _flush_pending_lsp_diagnostics"] R4 --> R5["⑤ 构建 MessageRequest"] R5 --> R6["⑥ turn_loop.run 流式请求"] R6 --> R7{取消?} R7 -->|是| RetC["return"] R7 -->|否| R8["⑦ append assistant_message"] R8 --> R9{有 tool_calls?} R9 -->|否| RetD["return 对话结束 ✓"] R9 -->|是| R10["⑧ _build_tool_use_message"] R10 --> R11["⑨ _execute_tool_calls"] R11 --> R12["⑩ extend tool_results"] R12 --> R13["⑪ run_post_tool_checkpoint"] R13 --> R14{工具出错?} R14 -->|是| R15["⑫ run_error_escalation_checkpoint"] R14 -->|否| R16["重置 consecutive_errors=0"] R15 --> R17 R16 --> R17{Plan + update_plan 成功?} R17 -->|是| RetP["return 停止"] R17 -->|否| Round end Round --> Limit["超过 100 轮 → ErrorEvent"]

每一轮 round 的 12 步详解

#步骤通俗解释细节
①Steer 注入 用户在中途又输入了？追加到消息列表 handle.drain_steers() 非阻塞取出 steer 队列，每条变成 Message.user
②容量预检 上下文快满了？提前压缩 run_pre_request_checkpoint：若决策为 TARGETED_CONTEXT_REFRESH → 调 _emergency_compact
③压缩触发 消息太多或 token 超阈值？总结旧消息 硬上限 500 条消息；token 阈值默认 500K floor；压缩后 summary 可能追加到 system_prompt
④LSP 刷新 把上一轮代码编辑的诊断信息塞给模型 _flush_pending_lsp_diagnostics → 合成一条 user 消息
⑤构建请求 打包 model、messages、tools、温度等参数 MessageRequest 含 reasoning_effort / temperature / extra_body
⑥流式单轮 调 LLM，实时收文字和工具调用 委托给 TurnLoop.run（见下一节）
⑦追加 assistant 把模型的回复加入历史 —
⑧⑨执行工具 模型要调工具？逐个或并行执行 见第 8 节
⑩追加 tool_results 工具结果加入历史，下一轮 LLM 能看到 —
⑪⑫容量后检 + 错误升级 工具执行后检查容量；连续出错则升级告警 step_error_count 累加；consecutive_tool_error_steps 追踪连续出错轮数
—Plan 停止 plan 模式下 update_plan 成功就停 should_stop_after_plan_tool

循环结束条件（任一满足即退出）：

  1. 模型返回纯文本，没有 tool_calls → 正常结束
  2. 用户取消 → cancelled=True
  3. Plan 模式下 update_plan 工具成功执行
  4. 达到 max_tool_round_trips（100）→ 发 ErrorEvent

透明重试

如果流式请求还没收到任何内容就失败了，会静默重试（最多 2 次），用户无感知。一旦已经收到部分内容，就不再透明重试，避免重复输出。

8.2 并行条件 should_parallelize_tool_batch

批次内每一个工具都必须满足：

  • read_only = True（只读，不改文件）
  • supports_parallel = True
  • approval_required = False（不需要审批）
  • interactive = False（不需要用户交互）

并行路径用 asyncio.gather 同时跑，但事件发射和结果组装仍是顺序的（保持顺序一致）。

8.4 辅助函数 _build_tool_use_message

把 LLM 返回的 ToolCall[] 转成一条 assistant 消息，内容是 ToolUseBlock[]（id + name + input）。这条消息和后续的 tool_result 消息配对，LLM 才知道「我调了什么、结果是什么」。

9. 审批流程 _handle_approval_flow

危险操作（如 exec_shell）执行前需要用户点头。返回值 True = 拒绝，False = 允许。

flowchart TD Start --> Cache["build_approval_key → approval_cache.check"] Cache --> Hit{缓存命中 APPROVED?} Hit -->|是| OK1["ApprovalResolvedEvent cached → return False"] Hit -->|否| Summary["填充 input_summary
MCP 用专用描述，其他用 _summarize_call_args"] Summary --> Emit["ApprovalRequiredEvent → TUI 弹窗"] Emit --> Wait["approval_handler.request_approval 阻塞等用户"] Wait --> Decision{用户决定?} Decision -->|DENIED| Deny["SandboxDeniedEvent → return True"] Decision -->|APPROVED| Allow1["缓存 + record → return False"] Decision -->|APPROVED_SESSION| Allow2["会话级缓存 + record → return False"]

_summarize_call_args — 审批摘要

帮审批对话框显示「具体在批什么」，而不是只显示工具名。优先级：

  1. prompt / message / objective（任务描述）
  2. command / cmd（shell 命令）
  3. path / file_path / source_path / dest_path（文件路径）
  4. content / text / input（内容）
  5. 兜底：任意非空参数的第一行

取首行，最长 200 字符。

审批缓存

用户选「本次会话都批准」(APPROVED_SESSION) 时，相同 fingerprint 的工具调用不再弹窗。APPROVED 只批准这一次。

11. LSP 诊断（代码编辑后自动检查）

三个函数配合，让 AI 知道「你刚改的文件有语法错误」：

函数何时调用做什么
_get_lsp_manager 内部用 从 tool_context.metadata[LSP_MANAGER_KEY] 取 LspManager
_run_post_edit_lsp_hook 工具成功执行后 找出被编辑的文件 → 读内容 → 调 LSP diagnostics_for → 结果存入 pending_lsp_blocks
_flush_pending_lsp_diagnostics 每轮 round 发 LLM 请求之前 render_blocks → 合成一条 Message.user 追加到 messages

设计原则：LSP 失败完全静默，绝不阻塞 agent。LSP 挂了？没关系，只是少了一条诊断信息。

12. 快照与 Undo

_take_pre_tool_snapshot

对 write_file、edit_file、apply_patch 三个写文件工具，执行前：

  1. 用 edited_paths_for_tool 找出要改的文件
  2. 读文件字节（≤1MB 才快照，超大跳过）
  3. 文件不存在则记 None（表示新建文件）
  4. 存入 tool_snapshots[tool_call_id]
  5. 最多保留 5 个快照，超出删最老的

失败不影响工具执行（best-effort）。

undo_last_tool

弹出最近一个快照，逐个文件恢复：

  • original=None → 删除文件（恢复「不存在」状态）
  • original=bytes → 写回原内容

返回 (success: bool, message: str)，供 /undo 命令使用。

13. 压缩与容量管理

Engine 有三层压缩/容量保护，防止上下文爆炸：

层级触发点函数说明
1. 常规压缩 每轮 round 开始前 should_compact + compact_messages_safe 消息 >500 条 或 token 超阈值（默认 500K floor）；用 LLM 总结旧消息，保留最近几条
2. 容量预检 每轮 round 开始前 run_pre_request_checkpoint CapacityController 观测 token/工具密度；决策 TARGETED_CONTEXT_REFRESH → _emergency_compact
3. 紧急压缩 TurnLoop 发现输入超预算 _emergency_compact TurnLoop 内部回调，最多尝试 3 次；仍超则报 CONTEXT_OVERFLOW

压缩后 system_prompt 变化

如果 compact 产生了 summary_prompt，会追加到 system_prompt 末尾：system_prompt + "\n\n" + summary_prompt，让模型知道之前聊了什么。

工具结果压缩

compact_tool_result_for_context：工具返回的内容可能很大（比如读了整个文件），塞进上下文前会按模型窗口做截断/压缩。

容量错误升级

run_error_escalation_checkpoint：工具连续出错时，CapacityController 会评估是否需要干预（当前 Python 版主要是日志，完整 VERIFY_AND_REPLAN 待实现）。

14. Cycle 归档（可选，默认关闭）

_maybe_advance_cycle — 超长会话的分段归档机制：

  1. 估算当前 messages 的 token 数
  2. should_advance_cycle 判断是否超过模型阈值（默认 ~768K）
  3. 超过 → archive_cycle 把整个对话写到磁盘 JSONL
  4. messages 替换为最近 8 条（seed），继续对话
  5. _cycle_n += 1，更新开始时间

失败不阻塞对话（best-effort）。需要 Config.cycle_enabled = True 才启用。

持久化 _auto_persist_session

每 turn 完成后，best-effort 写入 user_sessions_dir/current.json：

{
  "model": "...",
  "turn_counter": 42,
  "messages": [...]
}

先写 .tmp 再 atomic replace，失败静默忽略。
