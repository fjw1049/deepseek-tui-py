00 · 总览

先把整条链路装进脑子

Workflow 不是「再开一个聊天窗口」，而是：你交给它一张流程图说明书（JSON IR）， 运行时按图 spawn 子 agent，每步可落盘，还能隔离改代码、后台长跑。 

flowchart LR
  U["用户任务"] --> T{"workflow 工具"}
  T -->|name+task| C["找 JSON 说明书"]
  T -->|spec| C2["直接用你给的 IR"]
  T -->|run_id| R["从断点续跑"]
  T -->|detach:true| D["丢进 TaskManager"]
  C --> E["run_workflow 引擎"]
  C2 --> E
  R --> E
  D --> E
  E --> A["子 agent / fanout / loop"]
  A --> OUT["结果 + run_id"]
记忆口诀：说明书（IR）→ 引擎按步跑 → 每步可存档 → 可选隔离工作区 / 可选后台。
01 · 入口

模型（或你）怎么调用

工具名是 workflow。四种入口互斥（detach 可叠加在前三种上）：

你传什么 系统做什么 生活化例子
name + task 去目录里找现成 JSON，把 task 填进提示词 点外卖选「套餐 A」，备注写「少辣」
spec 用你现场写的完整流程图 自己画菜谱现做
run_id 读上次存档，跳过已完成步骤 游戏读档继续打 Boss
再加 detach: true 立刻返回任务号，后台继续跑 外卖下单后不用盯厨房

例子：最常见调用

// 用户说：「审查 engine / tools / workbench 集成风险」
{
  "name": "repo_review",
  "task": "审查 engine/tools/workbench 集成风险"
}
02 · 命名运行

四级发现：谁的说明书优先

写了 name: "repo_review" 之后，系统按优先级找同名 JSON（越靠前越优先）：

  1. <当前项目>/workflows/ — 团队共享
  2. <当前项目>/.deepseek/workflows/ — 本机项目私有
  3. ~/.deepseek/workflows/ — 你的全局私有
  4. 包内 presets — 自带的 repo_review / diff_review / spec_check
flowchart TD
  N["name = repo_review"] --> A{"项目 workflows/ ?"}
  A -->|有| W["用团队那份"]
  A -->|没有| B{".deepseek/workflows/ ?"}
  B -->|有| P["用项目私有"]
  B -->|没有| C{"~/.deepseek/workflows/ ?"}
  C -->|有| G["用用户全局"]
  C -->|没有| D["用内置 preset"]
为什么这样设计：模型不用每次现场编完整 IR（容易漏字段）；团队把稳定流程落成 JSON 共享。

对比：没有命名时的痛苦写法

// 以前：模型必须手写完整图，items 还得猜死
{
  "spec": {
    "phases": [{
      "steps": [
        { "type": "fanout", "items": ["engine", "tools", "workbench"], ... },
        { "type": "synthesis", ... }
      ]
    }]
  }
}
03 · 动态扇出

先看仓库，再决定开几路

Fanout 有两种填法，只能二选一：

items（静态）

写死名单：["engine","tools"]。适合你已经知道目标。

items_from（动态）

从上一步 structured 输出里抠数组。适合「先规划再并行」。

flowchart TD
  P["plan 子 agent"] -->|"返回 JSON"| J["{ targets: [engine, tools] }"]
  J --> F["fanout items_from path=$.targets"]
  F --> A1["子 agent: Inspect engine"]
  F --> A2["子 agent: Inspect tools"]
  A1 --> S["synthesis 汇总"]
  A2 --> S

例子：plan 返回 2 个 → 只开 2 路

// IR 片段（preset repo_review 里已有）
{ "id": "plan", "type": "agent", "prompt": "{{task}} → 产出 targets[]" }

{
  "id": "inspect",
  "type": "fanout",
  "items_from": { "step": "plan", "path": "$.targets" },
  "agent": { "prompt_template": "Inspect {{item}} for {{task}}" }
}

// 若 plan 返回 5 个 targets → 开 5 路（上限 16）
核心收益：「声明式编排」比手写多次 agent_spawn 更稳——扇出数量跟仓库实际情况走，不是猜。
04 · 模板与预置

同一份 JSON，换任务就能重跑

提示词里的占位符，运行时才填：

占位符含义例子
{{task}}本次用户任务「审查集成风险」
{{item}}fanout 当前项engine
{{round}}loop 第几轮2
{{outputs.plan}}某步文本输出synthesis 引用上游

三个内置套餐

repo_review

plan → items_from fanout → synthesis。多模块集成风险。

diff_review

plan lenses → fanout → synthesis。审当前 diff。

spec_check

抽需求 → fanout 对照实现 → report。偏只读。

policy 小抄

analysis_only 限制写工具；worktree: on 隔离改码；墙钟超时等。

05 · Loop

有界重试：修到 done 或到上限

不必上「动态 JS 控制器」。用 loop + until 就够做「再验证一轮」。

flowchart TD
  S["进入 loop refine"] --> R1["round=1 子 agent"]
  R1 --> C1{"structured.done == true ?"}
  C1 -->|是| STOP["提前结束"]
  C1 -->|否| R2["round=2"]
  R2 --> C2{"done ?"}
  C2 -->|是| STOP
  C2 -->|否| R3["round=3 = max"]
  R3 --> END["强制结束"]

例子

{
  "type": "loop",
  "id": "refine",
  "max_rounds": 3,
  "until": { "path": "$.done", "equals": true },
  "steps": [{
    "type": "agent",
    "prompt": "Round {{round}}. Fix remaining. Return {done, notes}."
  }]
}
// 第 2 轮返回 {"done": true} → 不再跑第 3 轮
06 · 存档

跑到一半断了：靠 run_id 续

每次运行会在项目下写： .deepseek/workflow-runs/<run_id>/run.json

里面记着：说明书快照、已完成 step id、各步 outputs、状态、可选 worktree 路径。

sequenceDiagram
  participant U as 用户/模型
  participant W as workflow 工具
  participant Disk as run.json
  U->>W: name+task 开始跑
  W->>Disk: 每完成一步 checkpoint
  Note over U,W: Esc / 超时 / 崩溃
  U->>W: { run_id: "wf_a1b2..." }
  W->>Disk: 读已完成 steps
  W->>W: 跳过完成项，从下一 step 继续
注意：这是「停了再启」，不是后台不停跑。要后台 → 看下一节的 detach。

例子

{ "run_id": "wf_a1b2c3d4e5f6" }
07 · Worktree

并行改代码时，别踩烂主目录

默认 policy.worktree: "off"。打开后，整次 run 的子 agent 都在独立 git worktree 里干活。

没有 worktree

3 个 fanout 同时改同一工作区 → 冲突、难回滚。

有 worktree

改动落在隔离分支；主目录 git status 干净；结束给人看 diff/PR。

flowchart TB
  subgraph Main["主工作区（保持干净）"]
    M["你的日常代码"]
  end
  subgraph WT[".deepseek/workflow-runs/wf_xxx/tree"]
    B["分支 deepseek-wf/wf_xxx"]
    A1["agent A 改测试"]
    A2["agent B 改实现"]
  end
  Main -.->|git worktree add| WT
  B --> A1
  B --> A2

规则（MVP）

  • 路径：.deepseek/workflow-runs/<run_id>/tree
  • 分支：deepseek-wf/<run_id>
  • 该 run 所有 spawn 的 workspace = 这个 tree
  • 跑完不自动删；结果里带回 worktree_path + branch
  • 不是 git 仓库却开了 on → 直接报错（fail-closed）
  • run_id 续跑 → 复用同一 tree
// 在 spec.policy 里打开
"policy": { "worktree": "on", "approval_mode": "trusted_workflow" }
08 · Detach

关电脑前也能丢给后台（进程还在时）

现在：tool call 断了 → 要人手动 run_id resume。
Detach：立刻返回，由 TaskManager 工人继续推同一 run_id 到终态。 

sequenceDiagram
  participant U as 主会话
  participant W as workflow 工具
  participant TM as TaskManager
  participant Disk as run.json
  U->>W: name+task, detach:true
  W->>Disk: create_run (+ 可选 worktree)
  W->>TM: 入队 [workflow-detach] run_id=...
  W-->>U: 立刻返回 run_id + task_id
  Note over U: 主回合可结束 / Esc 只停等待
  TM->>Disk: worker 驱动 run_workflow
  TM->>Disk: 直到 completed / failed / cancelled

例子

{
  "name": "repo_review",
  "task": "跑 deep 审查，我要去开会",
  "detach": true
}
// 立即返回：
// run_id=wf_...  task_id=task_...
// 取消：用 task_cancel，不是 Esc
边界：MVP 复用现有 TaskManager，不是独立 OS 守护进程。 TUI/server 进程退出后，工人也就停了。真正「关机还在跑」需要以后另做 daemon。
09 · 拼起来

一次「审查 + 隔离 + 可续跑」长什么样

flowchart TD
  U["用户：审查并自动修测试"] --> R{execution-router skill}
  R -->|简单| S["单 agent 搞定"]
  R -->|适合套餐| N["workflow name=repo_review"]
  N --> WT{"policy.worktree=on?"}
  WT -->|是| TREE["建 git worktree"]
  WT -->|否| MAIN["用主目录 cwd"]
  TREE --> PLAN["plan → targets[]"]
  MAIN --> PLAN
  PLAN --> FAN["fanout items_from"]
  FAN --> SYN["synthesis"]
  SYN --> LOOP{"要 refine?"}
  LOOP -->|loop until| FIX["最多 3 轮修"]
  LOOP -->|不用| DONE["ToolResult + run_id"]
  FIX --> DONE
  DONE -->|中断| RES["run_id 续跑同 tree"]
  DONE -->|detach| BG["TaskManager 推到终态"]

技能怎么掺一脚

  • execution-router：先判断「单 agent / 用哪个 named workflow / 要不要新建 IR」，避免无脑 fan-out。
  • workflow-guide：教模型写 DeepSeek IR（不是 Pi 那套 artifact-graph）。
