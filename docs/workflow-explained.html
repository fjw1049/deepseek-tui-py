deepseek-tui · Workflow

整段编排，一次工具调用跑完

主模型不用手写十次 agent_spawn。它交一份结构化计划（Workflow IR）， 运行时按 DAG 调度子代理，实时推进度，最后把结果塞回主对话。

◇
一句话类比：项目经理 + 施工队

主模型是「甲方」：说清要干什么。workflow 工具是「项目经理」：拆任务、盯进度、合并汇报。 每个 step 派出去的子代理是「施工队」。经理不会把整栋楼交给甲方自己砌砖——编排在一次工具调用里做完。

端到端 五种入口 九种步骤 调度引擎 Dynamic 策略与取消 模板变量 断点恢复 进度 UI 实战例子 易踩坑

1. 一次调用，从头到尾发生了什么

核心不在「魔法」，而在一条清晰的调用链。

你说话，模型决定要编排

例如：「用 workflow 多视角审查 engine / tools / workbench」。

模型调用 workflow 工具

传 name+task、spec、mode=dynamic 或 run_id。工具需审批一次（整段编排）。

Engine 注入取消与进度通道

挂上 engine_cancel_event、workflow_emit、workflow_tool_call_id，进度才能 upsert 到同一张卡片。

解析 IR → 编译成 DAG

v1 的 phases 会串成链；v2 的 graph 直接用节点与边。环检测失败立刻报错。

ready-set 调度器开工

schedule_workflow() 找出「前驱都结束」的节点，按 concurrency 一批批跑。

每个 agent 步 → 子代理 spawn + wait

DeepSeekAgentRunner 走 SubAgentManager，进程内跑，不新起 deepseek-tui 进程。

每完成一步就 checkpoint + 推进度

落盘到 .deepseek/workflow-runs/<run_id>/，SSE 发 workflow.progress，Workbench 刷新树。

汇总结果，回到主模型

优先取最后一个 synthesis/reduce 的 structured；否则给各步 preview。主回合继续，不走 subagent handoff。

flowchart TB U["用户消息"] --> M["主模型 LLM"] M --> T["workflow 工具"] T --> P["解析 / 校验 IR"] P --> G["编译 CompiledGraph"] G --> S["ready-set 调度器"] S --> R["DeepSeekAgentRunner"] R --> SA["SubAgentManager.spawn"] SA --> CP["checkpoint + progress SSE"] CP --> S S --> OUT["ToolResult + snapshot"] OUT --> M2["主模型继续对话"]

2. 五种入口（互斥）

一次调用只能选一种「怎么开始」；detach 可叠加。

name + task

命名工作流

从 catalog 加载 JSON。推荐：repo_review、diff_review、spec_check、adaptive。

mode: dynamic

自适应糖

内置单节点 dynamic 控制器。适合目标清晰、路径未知的任务——让控制器边跑边长图。

spec

临时 IR

模型手写 v1 phases 或 v2 graph。最灵活，也最容易写错依赖。

run_id

断点续跑

从磁盘恢复 outputs / 已完成步 / runtime_graph。不能和 name/spec 混用。

detach: true

后台挂起

立刻返回 run_id + task_id，由 TaskManager 跑完。进度不走 Workbench 实时 SSE。

Catalog 查找顺序

<cwd>/workflows/ → <cwd>/.deepseek/workflows/ → ~/.deepseek/workflows/ → 包内 presets/（后者优先级最低，可被本地同名覆盖）

3. 九种步骤：各自在干什么

用「工地」类比记语义；真正执行在 scheduler.py。

agent
派一个工人做一件事

固定 prompt，spawn 一次子代理。可带 output_schema 拿结构化结果。

fanout
同一工种，并行扫多个目标

items 固定列表，或 items_from 从上一步 structured 数组展开。每项输出键 id:item。

pipeline
同一目标，多道工序串行

每个 item 按 stages 顺序跑；下一 stage 用 {{previous}} 接上文。

synthesis
写总结报告

用模板引用 {{outputs.*}}。ready-set 默认允许 partial join（部分前驱失败也能跑）。

reduce
明确等谁，再汇总

像 synthesis，但声明 from。默认 source_policy: partial；设 success 则任一源失败就炸。

loop
同一套子步骤最多转 N 轮

max_rounds≤8。可选 until（JSON path 等于某值就停）。体内不能再套 loop。

dag
嵌套小图

子图自己做 ready-set。无边时默认串成链。output_from 指定向上冒泡哪一节点。

dynamic
现场调度员边跑边改图

控制器每轮输出动作：spawn / fanout / reduce / replan / stop… 改完图再 drain 一轮。

support
纯函数小帮手（不叫 LLM）

白名单：dedupe_findings、flatten_previews、merge_json。

4. 调度引擎：谁先跑、谁等谁

真正干活的是 schedule_workflow()，不是线性 for 循环。

节点何时变成「可跑」？

节点类型前驱条件
普通 agent / fanout / … 每个前驱都在 completed 或 skipped
reduce（partial）/ synthesis 前驱都「终态」（完成/失败/跳过），且至少一个 completed
reduce（success） ready 后执行时若源失败 → 抛 WorkflowFailedError

一批最多跑几个？

每次取 ready 列表的前 policy.concurrency 个，asyncio.gather 并行。某步失败且 fail_fast 时，会取消同批孤儿任务 + 所有已 spawn 子代理。

fanout 内部还有自己的并发上限：min(step.concurrency, policy.concurrency)。全局 spawn 数还受 max_agents 限制。

两种写法，一种执行内核

版本作者写法编译结果
v1 phases[].steps[] 按全局顺序串成一条链（phase 之间也不能并行）
v2 graph.nodes + edges / after 真正的 DAG；reduce/support 的 from 会自动补边
flowchart LR A["plan
agent"] --> B["inspect
fanout"] B --> C["final
reduce"] A -.->|outputs.plan| C B -.->|outputs.inspect| C

上图即内置 repo_review：计划 → 并行巡检 → 汇总。

5. Dynamic：现场改图纸的调度员

静态图写死路径；dynamic 让控制器每轮决定「下一步加什么节点」。

拼控制器 prompt

当前图状态、已有 outputs 索引、预算余量 → 喂给 controller agent。

结构化决策

按固定 schema 输出动作列表；校验合法性、防 stall（连续相同决策签名会失败）。

改图 apply_decision_actions

spawn / fanout / reduce / support / splice_dag / nested_workflow / replan / synthesize / stop。

drain 一轮 ready 节点

本轮新加的可跑节点全部跑完，再进入下一轮决策。不是「改完立刻再决策」。

stop 或 synthesize

synthesize 隐式 stop；最后再 drain 一次，交出终态。

控制器动作速查
动作效果批次规则
spawn加一个 agent 节点可与工作动作同批
fanout加并行展开；受 max_fanout_items同上
reduce加 join 汇总节点同上
synthesize加终态汇总并隐式 stop必须本批最后且唯一
support加纯函数节点可同批
splice_dag批量插节点+边可同批
nested_workflow嵌套整段 workflow受嵌套深度预算
replan丢掉未完成的生成节点不能与工作动作同批
stop结束；success=false 则失败必须独占一批
预算墙

Dynamic 另有独立预算：决策轮数、突变次数、fanout 项数、嵌套深度、墙钟。 突变耗尽后只允许 stop / synthesize / replan。默认控制器 allow_write_tools: false。

6. 策略、审批、取消、超时

这些旋钮决定「子代理敢不敢写文件」以及「出事怎么办」。

字段默认通俗含义
approval_mode trusted_workflow analysis_only：强制只读工具； trusted_workflow：子代理自动批准； strict：写操作仍要审批
on_error continue 继续：失败步记 null，partial 汇总还能跑；fail_fast：立刻停全场
max_agents 10（上限 32） 同时「在编」的子代理上限（含预留槽）
concurrency 4 ready-set 每批并行节点数
wall_clock_seconds 600 整段墙钟超时；超时写 stop-intent 并取消已 spawn
token_budget 无 用字符数/4 估算；超限失败
worktree off on 时在独立 git worktree 里改代码，跑完不自动删
取消有三层

① 内存里的 engine_cancel_event（Esc / 回合取消）→ ② 磁盘 stop-intent.json（跨进程也能看见）→ ③ 对每个 spawned_agent_ids 调 manager.cancel()。 调度器每步都会先查 stop-intent。

7. 模板变量：上下文怎么灌进 prompt

运行时渲染，不交给模型自己拼字符串。

变量用途
{{task}}本次运行任务正文
{{item}}fanout / pipeline 当前项
{{round}}loop 当前轮次
{{previous}}pipeline 上一 stage 的 preview
{{outputs.X}}步骤 X 的 preview（默认截断约 2000 字）
{{outputs.X.full}}完整 text（硬上限 32KB）
{{outputs}}所有步的一行索引，不是全文

items_from.path 只支持极简 JSONPath：$、$.a.b、$.a[0].b。

8. 断点、租约、后台 detach

跑到一半断电？留下 run_id 就能接着干（同步路径）。

checkpoint

写什么

completed / skipped / failed、outputs、snapshot、runtime_graph、dynamic_states、token 估算…

lease

防双开

fcntl.flock + 90s 心跳。活跃 lease 时 resume 会被拒绝。

resume

怎么续

跳过已有输出的 step，恢复动态图袋。已完成 run 不能再 resume。

实现差异（重要）

同步 WorkflowTool.execute 的 checkpoint 会持久化 runtime_graph 与 dynamic 状态。 Detach 路径当前不会完整写入这些字段，因此 dynamic 工作流经后台中断后，resume 可能无法完整恢复可变图。

9. 进度怎么画到界面上

同一张 workflow 卡片靠 tool_call_id 做 upsert。

10. 实战：repo_review 怎么跑

内置预设，最适合建立直觉。

调用形态

{
  "name": "workflow",
  "arguments": {
    "name": "repo_review",
    "task": "审查 engine、tools、workbench 的集成风险"
  }
}

plan（agent）

探索型子代理返回 {"targets":[...]} 结构化列表。

inspect（fanout + items_from）

从 $.targets 展开，最多 4 路并行巡检，每项写一份风险报告。

final（reduce, partial）

等 plan + inspect，合成 ok / verdict / findings。部分巡检失败也能汇总。

11. 审核结论：别踩这些坑

对照当前代码的边界，不是文档愿望清单。

调度与图语义
  • v1 phases 会编译成全局串行链——跨 phase 不会并行。
  • loop 体内的 step 不进顶层 DAG，由 loop 自己迭代；每轮会清掉 body 相关 outputs。
  • fanout 有双键：节点级合并 preview，明细在 {step}:{item}。
  • dag 内不能再嵌 dag/dynamic；loop 不能嵌 loop。
取消、超时与子代理
  • 单 agent 未设 timeout_seconds 时默认等 1 小时。
  • 整段超时有约 5 秒 grace；若刚好跑完则不会误标 timed_out。
  • agent 失败返回 None（不是抛到主回合），由 on_error 决定命运。
Detach / Resume
  • Detach 没有 orchestrator 注入的 workflow_emit → 无 Workbench 实时进度树。
  • Detach checkpoint 对 dynamic 可变图状态不完整（见上）。
  • Resume 要求 step 既在 skip 集合又在 outputs 里，否则会重跑。
Dynamic 特有
  • 动作批次有硬规则：stop 独占；synthesize 必须最后；replan 不与工作动作混批。
  • 连续相同决策签名 → stall 失败，防止死循环。
  • 缺省 after 时新节点可立即 ready——控制器必须自己想清楚依赖。
